# kizu v0.2 — scar + 背景色 diff + マルチエージェントフック層 (post-adversarial-review)

この ExecPlan はリビングドキュメントです。`Progress` / `Surprises & Discoveries` / `Decision Log` / `Outcomes & Retrospective` の各セクションは作業進行に合わせて継続的に更新します。

## Purpose / Big Picture

このマイルストーンを完了すると、ユーザーは Claude Code / Cursor / Codex CLI / Qwen Code / Cline / Gemini CLI と併走させながら、kizu から **キー一打で scar（`@kizu[ask|reject|free]:` インラインコメント）を刻み、エージェントに（エージェントが対応する範囲で）自動フィードバック**できるようになります。

エージェントごとの統合深度はバラつきがあります:

| **エージェント** | **PostToolUse 即時通知** | **Stop 最終チェック** | **備考** |
|:---|:---:|:---:|:---|
| Claude Code | ✅ | ✅ | 基準実装 |
| Cursor 1.7+ | ✅ | ✅ | `afterFileEdit` + `stop`、production |
| Qwen Code | ✅ | ✅ | Claude Code 互換 |
| Codex CLI | ⚠️ Bash matcher 限定で不可 | ✅ | Stop のみ有効 |
| Cline v3.36+ | ✅ (best-effort) | ❌ | **Stop 相当 hook が無い**。parity 非保証 |
| Gemini CLI | ❌ | ❌ | stream-json consumer 経由の書き込み側のみ |
| Windsurf | ⏸ | ⏸ | **v0.2 スコープ外**（スキーマ検証待ち、v0.2.1 以降） |

具体的な観察可能挙動:

- kizu を起動中に `a` を押すと、選択中 hunk の直上に言語のコメント構文で `// @kizu[ask]: explain this change` が書き込まれる。`r` は `// @kizu[reject]: revert this change`、`c` はミニ入力欄→任意テキスト (`// @kizu[free]: <ユーザー入力>`)
- `x` を押すと選択中 hunk が `git checkout -p` 相当で revert される (scar は挿入しない)
- `e` を押すと `$EDITOR +<line> <file>` が起動し、該当行にジャンプしたエディタが開く。終了後に kizu に戻る
- `Space` で「見た」マーク（TUI 内部状態のみ、ファイルには何も書かない）
- **diff 表示が delta 風の背景色ベースに刷新される**: `+`/`-` の prefix マーカー列が廃止され、追加行は緑の背景、削除行は赤の背景で全幅塗りつぶし表示になる
- `kizu init` を実行すると、PATH 上の AI コーディングエージェントを自動検出してインタラクティブ multi-select UI が表示され、選んだエージェントそれぞれの設定ファイルに PostToolUse/Stop hook を**既存設定とマージしながら**書き込む。各エージェントの対応範囲は UI 上に明示表示する (`Stop only` / `PostTool best-effort` 等)
- `kizu init --agent cursor,codex --non-interactive` のように CI 用に飛ばせる
- Claude Code / Cursor / Qwen Code / Cline が scar のあるファイルを編集しようとすると、`kizu hook-post-tool` が `additionalContext` で「未対応 scar が N 件あります」と通知する
- **Stop gate が効くエージェント (Claude Code / Cursor / Codex / Qwen Code)** ではターン終了時に未対応 scar があると `kizu hook-stop` が exit 2 で stop を阻止し、stderr にファイル名と行番号が出力される
- **Stop gate のスキャン範囲は tracked + untracked の両方**（`git diff --name-only` + `git status --porcelain --untracked-files=all` を合成）。新規作成ファイルに刻んだ scar も検出される
- Gemini CLI 併用時は `gemini -p "..." --output-format stream-json | kizu consume-gemini-stream` で tool_use/tool_result を tail し、scar は書き込み側のみサポート（逆方向フィードバックなし）
- `kizu teardown` で全エージェントの hook 設定を一括除去する（選択 UI なし、検出した全てから外す）
- PostToolUse イベントログは **sanitize されたメタデータのみ**を `$XDG_STATE_HOME/kizu/events/` (Linux) / `~/Library/Application Support/kizu/events/` (macOS) に `0700` パーミッションで保存。`tool_input` / `tool_response` の生本文は保存しない。TTL 24 時間で自動削除、上限 1000 エントリ（v0.3 ストリームモードのメタデータ源）

scar まわりの非同期 feedback パターンの背景は `docs/inline-scar-pattern.md`、Claude Code hook の落とし穴は `docs/claude-code-hooks.md`、マルチエージェント調査結果は `deep-research-ai-agent-hooks.md` に一次情報が集約されています。このマイルストーンはこれらを Rust 実装に落とす作業です。

v0.3 スコープ（`--attach` でのターミナル分割、`/kizu` スラッシュコマンド、ストリームモード UI、`~/.config/kizu/config.toml`、Windsurf installer、Aider `--lint-cmd` 近似統合）はこの ExecPlan には含めません。v0.2 完了時点では `<state_dir>/events/` への sanitized メタデータ書き込みまで完了し、**UI 側の可視化は v0.3 に持ち越し**ます。

## Surprises & Discoveries

実装中に発見した予期しない挙動、バグ、最適化、性能トレードオフを証拠とともに記録します。

### 着手前に既知の landmine（deep-research-ai-agent-hooks.md より先行記録）

- Observation: OpenAI Codex CLI の `PreToolUse` / `PostToolUse` matcher は現時点で **Bash ツール限定**で、Write/Edit 系ツールには刺さらない。
  Evidence: `developers.openai.com/codex/hooks` の "Matcher Patterns" セクションで `PostToolUse / PreToolUse: tool name (e.g., "Bash")` と明記。
  Impact: Codex 統合では `hook-post-tool` を諦め、`Stop` hook + git diff grep の最終チェックだけに絞る。M7 の `CodexInstaller` は file edit 用 hook を設置せず、Stop hook のみ書き込む。
- Observation: Cursor の hook は **production ステータス**で、`CLAUDE_PROJECT_DIR` 環境変数まで exposed している。
  Evidence: `cursor.com/docs/hooks` の Environment Variables 表に `CLAUDE_PROJECT_DIR: Alias for project dir` とある。
  Impact: Cursor と Claude Code は hook コマンド側のコードパスをほぼ共有できる。`CursorInstaller` は Claude Code 相当の JSON を `.cursor/hooks.json` (version: 1) に書くだけで良い。
- Observation: Cline は hook を **ファイル配置型**で設定する（`.clinerules/hooks/PostToolUse` というファイル名そのものが event binding）。
  Evidence: `cline.ghost.io/cline-v3-36-hooks/` の "Types of Hooks" セクション。
  Impact: `ClineInstaller` だけ他と実装形態が違う。JSON マージではなく、実行可能スクリプトを 1 本配置する形になる。kizu 本体を呼ぶ shell wrapper を生成する。
- Observation: Cline は **macOS / Linux のみ**対応、Windows 非対応。
  Evidence: 同上 "Platform Support" セクション。
  Impact: `ClineInstaller` を Windows でビルドはするが、install 時に `cfg(not(target_os = "windows"))` でガードするか、warning を出して skip する。
- Observation: Gemini CLI は hook 機構が無く、`--output-format stream-json` の JSONL stream tail が唯一の統合点。
  Evidence: `geminicli.com/docs/cli/headless/` の stream-json ドキュメント。
  Impact: `GeminiStreamInstaller` は設定ファイルを触らず、`kizu consume-gemini-stream` 使用方法を標準出力に表示するだけ。双方向フィードバックは v0.2 スコープ外。
- Observation: Roo Code / Zed / Continue.dev は現時点で hook 機構を持たない（Roo Code は issue #11504, #12025 で申請中）。
  Evidence: 各公式ドキュメント + GitHub issue 確認済み。
  Impact: v0.2 の自動検出には含めない。将来 hook が追加されたら ExecPlan v0.3 で取り込む。
- Observation: v0.1 の素直な hook-stop 設計（`git diff --name-only | xargs grep -ln '@kizu\['`）は **untracked ファイルを取りこぼす**。
  Evidence: `git diff --name-only` の man page、および `src/git.rs` で既に `git status --porcelain --untracked-files=all` を使って untracked 合成している事実の交差参照。
  Impact: AI エージェントが新規作成したファイルに刻んだ scar は Stop gate を素通りする。これは AI の最頻出 action に対する穴で、critical 扱い。M5 で `enumerate_repo_files_with_scars` を導入して tracked + untracked を合成する。Codex adversarial review #2 で指摘された事項。
- Observation: `/tmp` への raw hook JSON 保存は **他 local user から読める** + 本文に秘密情報が混ざりうる。
  Evidence: POSIX デフォルトの `/tmp` permission (`1777`) と、Claude Code hook の `tool_input.content` / `tool_response.output` に edit したファイル本文がそのまま載る事実。
  Impact: v0.2 のまま ship すると情報漏洩パスを作る。M5 で sanitize + private state dir + 0700 permission + TTL を 3 点セットで導入。Codex adversarial review #1 で指摘された事項。
- Observation: Cline は Stop 相当 hook を持たない（PreToolUse / PostToolUse / UserPromptSubmit / TaskStart のみ）。
  Evidence: `cline.ghost.io/cline-v3-36-hooks/` の "Types of Hooks" セクション、deep-research 結果で再確認済み。
  Impact: Cline を「Claude Code parity」と宣伝するのは false safety。PostToolUse wrapper のみ配置する best-effort 統合にとどめ、ユーザーに対して明示的に warning を出す必要がある。Codex adversarial review #3 で指摘された事項。
- Observation: M1 実装中に、v0.1 の素直な `PollWatcher + compare_contents=true + Recursive(worktree)` 設計が起動時 `target/` 42,647 files / 1.3 GB の initial content scan で **3.5 秒フリーズ**することが発覚した。
  Evidence: `KIZU_STARTUP_TIMING_FILE` 計測で `watcher::start` が単独で 3.515 秒、他の startup step は合計 42ms 未満。
  Impact: v0.1 時代から潜んでいた UX regression。ADR-0015 で `target` / `node_modules` 等 15 種の top-level dir を hardcoded exclusion list で skip し、worktree root は `NonRecursive` + top-level children の `Recursive` watch の 2 層構成に変更して解決。ADR-0005 の「kizu 側で .gitignore filter を持たない」は維持。修正本体は Codex rescue で実装 (145 → 147 Rust test + 新規 `worktree_watcher_skips_target_directory` / `worktree_watcher_still_sees_root_level_file_writes` 各 1 本)。
- Observation: M1 の初期実装で `Modifier::REVERSED` を focused hunk の body style に載せたところ、REVERSED が bg/fg を swap してコードテキストが `Rgb(10, 50, 10)` の濃い緑で描画され、読めなくなった。
  Evidence: `selected_hunk_diff_lines_render_at_full_color` 相当の手動確認で fg が BG_ADDED 値に swap されていることを観察。
  Impact: ExecPlan 起草時の「REVERSED で cursor 行を反転」戦略は不採用に。左バー (`▎`/`▶`) と DIM コントラストで focus を伝える方針に切り替え (ADR-0014 で明記)。
- Observation: M1 の最初の drop では `is_selected` を無視して全 add/delete 行を uniform `bg` / 全 context 行を `DarkGray + DIM` にしたところ、focused hunk の context コードまで暗くなり「差分以外が暗すぎる」UX regression が発生した。
  Evidence: ユーザーからの直接フィードバック (2026-04-15 実機)。
  Impact: v0.1 の focused/unfocused コントラスト (focused = bright, unfocused = DIM) を復活。context 行も focused なら端末既定色、unfocused なら dark-gray + DIM に戻す。ADR-0014 の Decision 4 に最終ルールとして記載。
- Observation: Windsurf の file-edit hook 名は公式ドキュメントで未確定。
  Evidence: `docs.windsurf.com/windsurf/cascade/hooks` を調べた deep-research セクションで「実装前に再確認」と明示、file edit 特化 event 名が出てこない。
  Impact: 推測スキーマで `.windsurf/hooks.json` を書き込むと、既存 hook を silently 破壊するか完全 no-op になる。v0.2 では installer 自体を作らず、`AgentKind::Windsurf` を feature-gate する。Codex adversarial review #4 で指摘された事項。

## Decision Log

Record every decision made while working on the plan.

### 着手前に決定済み

- Decision: diff 表示を `+`/`-` prefix ではなく背景色（delta 風）に全面移行する
  Rationale: 視線移動のコストが下がり、シンタックスハイライト未実装でも視覚的情報量が増える。syntect の lazy-load は v0.1.1 で棚上げしていたが、背景色ベースに切り替えるとハイライトなしでも十分見やすくなるため、syntect 導入自体を v0.3 以降へ先送りする
  Date/Author: 2026-04-15 / Initial implementer
- Decision: `e` キーで `$EDITOR +<line> <file>` を spawn して外部エディタを開く
  Rationale: scar 挿入では表現しきれない複雑な介入（hunk 全体の書き換え、別ファイルの参照）を kizu から離脱せずに行えるようにする。`$EDITOR` は POSIX 的な最低共通項
  Date/Author: 2026-04-15 / Initial implementer
- Decision: マルチエージェント hook 層は **Claude Code 形式を正規系**として、他エージェントは shim で吸収する
  Rationale: Cursor / Qwen Code / Cline がほぼ Claude Code 形式をコピーしており、今後追加される後発エージェントもこの形式に収束する見込みが高い。内部データ構造を `NormalizedHookInput` で一本化すれば `hook-post-tool` 本体は 1 コードパスで済む
  Date/Author: 2026-04-15 / Initial implementer
- Decision: 先行統合対象は Claude Code / Cursor / Codex CLI / Qwen Code / Cline / Gemini CLI の **6 種**。Windsurf / Roo Code / Zed / Continue.dev / Aider は v0.2 では対応しない
  Rationale: Roo Code / Zed / Continue.dev は hook 機構を持たない。Aider は `--lint-cmd` 近似統合が exit code 意味論のズレで ExecPlan を汚すため v0.2.1 以降。**Windsurf は公式ドキュメントの file-edit hook 名が未確定**で、同じ deep-research 内でも「実装前に再確認」とマーク済み → 推測スキーマで書き込むとユーザーの editor 設定を静かに壊す可能性があるため v0.2 では外す（v0.2.1 以降で schema 検証 + fixture test を整えてから追加）
  Date/Author: 2026-04-15 / Initial implementer (ref: deep-research-ai-agent-hooks.md, Codex adversarial review #4)

## Context and Orientation

### 現在の状態（v0.1 MVP 完了時点）

kizu はカレントディレクトリの git worktree を監視する単一バイナリの TUI です。ファイル構成:

- `src/main.rs` (46 行) — `#[tokio::main(flavor = "current_thread")]` で `app::run` を呼ぶエントリポイント
- `src/app.rs` (4082 行) — TUI 状態機械、`App` 構造体、`handle_key`、watcher イベント統合
- `src/git.rs` (1445 行) — git CLI shell out、`compute_diff`、`parse_unified_diff`、`find_root`、`head_sha`、`git_dir`
- `src/watcher.rs` (1138 行) — `notify-debouncer-full` ラッパ、worktree 300ms / git_dir 100ms デバウンス
- `src/ui.rs` (1699 行) — ratatui フルスクリーン縦巻物 + popup picker

v0.1 では `a` / `r` / `c` / `x` / `e` の key dispatch は未実装。`a` `r` `c` は現在 picker の query 入力として吸収されるため、通常モードでは no-op。`src/app.rs` の `handle_key` 関数に 2 モード dispatch (通常 / picker) があるので、通常モード側で先に scar キーを捕捉する必要があります。

**重要**: v0.1 では **`Space` キーが picker の起動トリガ**に割り当てられています。v0.2 では scar の「見た」マーク (SPEC.md 既定) に `Space` を使うため、picker トリガを `Space` → `s` に付け替えます。`s` はまだ未使用のキーで、単打で picker をポップアップさせる既存意味論は保たれます。

同時にカーソル移動系のキー配置も変更します。v0.1 の現状は以下:

- `j`/`k` (および `↓`/`↑`): adaptive `next_change` / `prev_change` — 長い hunk 内では chunk scroll、短い hunk では hunk 単位の collapse 移動
- `J`/`K`: 厳密な `next_hunk` / `prev_hunk` — hunk サイズに関わらず次の hunk header へ飛ぶ

v0.2 ではこれを以下に変更します:

- `j`/`k` (および `↓`/`↑`): **据え置き**（adaptive `next_change` の挙動は触らない）
- `J`/`K`: **visual row 1 行単位のカーソル移動**に作り替え（新規追加メソッド）
- `h`/`l`: 現行の `J`/`K` の hunk jump 挙動を引き継ぐ

`J`/`K` を 1 行移動に落とすのは、scar 挿入時に「この行」を精密に指し示したいためです。adaptive 移動は巻物 navigation には最適ですが、scar は行単位の punctuated 操作であり、別キーでの単行移動が必要になります。`h`/`l` は vim の水平移動から転用する形ですが、kizu は縦巻物 UI のため水平移動は存在しないので、空いた 2 キーを hunk jump に転用します。

baseline SHA は起動時の `git rev-parse HEAD` で取得し、`App.baseline` に保持されます。初回コミットがない repo では空 tree SHA (`4b825dc642cb6eb9a060e54bf8d69288fbee4904`) にフォールバックします。`R` キーで再取得可能。

### 専門用語の定義

- **scar**: AI コーディングエージェントに対する非同期レビューとしてソースファイル内に直接書き込まれる `@kizu[ask|reject|free]: ...` コメント。コードコメント構文で書かれ、エージェントは次の Read/Edit でこれを認識する。bracket 部分の tag で category が一意に grep できる。`docs/inline-scar-pattern.md` 参照
- **hunk**: unified diff の 1 連続した変更ブロック。`@@ -L,S +L,S @@` ヘッダで始まる単位
- **PostToolUse hook**: AI エージェントが Write/Edit ツール実行直後に呼ぶ外部コマンド。stdin に JSON を流し込み、exit code と stdout でエージェントに feedback を返す
- **Stop hook**: AI エージェントがターンを閉じようとする瞬間に呼ぶ外部コマンド。exit 2 で stop を阻止すると次ターンに stderr 内容が指示として渡される
- **additionalContext**: Claude Code hook の出力 JSON フィールド (`hookSpecificOutput.additionalContext`)。次のターン冒頭にエージェントへ渡される追加コンテキスト文字列
- **stop_hook_active**: Stop hook 入力 JSON のフィールドで、これが true のときに Stop hook が exit 2 を返すと無限ループになるため、kizu 側で true を見たら即 exit 0 しなければならない
- **NormalizedHookInput**: kizu 内部でエージェント間差分を吸収した正規化済みデータ構造。`session_id` / `tool_name` / `file_paths` / `cwd` / `stop_hook_active: bool` を持つ
- **InputMode**: `App.input_mode: InputMode` の取る値。`Normal` / `Picker` / `ScarComment` / `Search { direction }` の 4 種（v0.2 で `ScarComment` と `Search` を追加）
- **smart case**: 検索 query が全小文字なら case-insensitive、大文字を 1 文字でも含めば case-sensitive として扱う matching 戦略。vim の `'smartcase'` と同じ意味論
- **file view**: `Enter` で遷移する zoom-in モード。1 ファイル全体を描画し、hunk 行には `BG_ADDED` / `BG_DELETED` を残したまま、diff に含まれていなかった行も context として表示する。`Enter` / `Esc` で通常モード (scroll view) に戻る。`App.input_mode::FileView { file_idx, cursor_line, scroll }` で保持

## Plan of Work

### M1（delta 風背景色）を最初に置く理由

UI の全幅レイアウトに手を入れる変更は scar 機能とは独立に完結するため、まずこれを終わらせて M2 以降と干渉しないようにします。`src/ui.rs` の `render_scroll_line` / `render_diff_row` で `+`/`-` prefix 列を組み立てている箇所を特定し、その列の幅を 0 に、代わりに `Line` 全体の `Style.bg` を `LineKind::Added → Color::Rgb(10, 50, 10)` / `LineKind::Deleted → Color::Rgb(60, 10, 10)` に設定します。背景色は dim 寄りの値にしてテキストとのコントラストを維持します（delta の `--dark` プロファイル参考値）。

visual cursor の選択行表示は現在 `Modifier::REVERSED` で実装しているので、そのまま流用可能です。REVERSED は fg/bg を swap するため、背景色ベースでも正しく反転します。

行折り返し時の背景色尾引きは ratatui の既知の挙動で、`Line` にした時点で末尾に到達したら背景色が途切れます。`Paragraph` の背景は行ごとではなく Block ごとに指定されるため、折り返し行も背景色を継続させるには `pad_line_to_width` のようなヘルパで末尾スペースを埋める必要があります。M1 のユニットテストでこの尾引きを検証する行を追加します。

### M2-M3（scar のコア）を pure function で切る

`src/scar.rs` を新設し、副作用の無い `detect_comment_syntax` と `render_comment` を最初に書きます。TDD の Red を極力長く取り、`compute_diff` が純粋関数だった v0.1 と同じ戦略を踏襲します。`insert_scar` だけは `std::fs` を触るので、テストは `tempfile::tempdir` で実 FS を使う integration style にします。

### M4（App へのキー統合）

`src/app.rs` の `handle_key` に通常モードの scar dispatch を挿入する場所は、既存の `j` / `k` / `R` / `q` ブロックの直前です。picker モード中は従来通り query 入力として吸収されるので、`InputMode::Picker` の case は触りません。

`c` の入力欄は新しい `InputMode::ScarComment` を追加して実装します。`ui.rs` のフッタ描画に 1 行追加し、`input_mode == ScarComment` のとき `> <current_text>_` を出します。`Enter` で確定、`Esc` でキャンセル、`Backspace` で 1 文字削除、それ以外の char は append。

`e` のエディタ起動は生々しい端末制御を伴います。`ratatui::DefaultTerminal` の `draw` を止めて、`crossterm` で alt screen を leave + raw mode を disable、`tokio::process::Command::new(editor).arg(format!("+{line}")).arg(path).status().await?` を await、復帰後に再度 alt screen enter + raw mode enable + `terminal.clear()?` で再描画します。これを `App::open_editor(&mut self, terminal: &mut Terminal) -> Result<()>` として切り出します。

`x` の hunk revert は git CLI shell out で行います。`git.rs` に `revert_hunk(root: &Path, patch: &str) -> Result<()>` を追加し、`git apply --reverse --` に hunk 本文を stdin で流し込みます。hunk 本文は `App.layout` の現在カーソル位置から逆算します。

### M5（hook サブコマンド）

`src/hook.rs` を新設し、`main.rs` の clap subcommand dispatch から呼び出します。全 hook サブコマンドの構造:

1. stdin を `serde_json::from_reader` で `serde_json::Value` に読み取り
2. `AgentKind` を `--agent` フラグから取得（default: `claude-code`）
3. エージェント別のフィールド名で `file_paths` を抽出（Claude Code は `tool_input.file_path`、Cursor は `tool_input.file_path`、Qwen Code は Claude Code 互換、Cline は `tool_input.path`、Windsurf は要再確認）
4. `scan_scars(&file_paths)` で未対応 scar を列挙
5. dedupe state を `/tmp/kizu-scars/<session_id>.json` から読み、既通知分を除外
6. エージェント別の envelope で JSON を stdout に出力、exit code を決定

`hook-log-event` は最もシンプルで、stdin をそのまま `/tmp/kizu-events/<ts>-<tool_name>.json` に atomic write するだけです。ファイル名衝突時は `-<nanos>` サフィックスで回避します。

### M6-M7（`kizu init` インタラクティブ）

`detect_agents` は `std::env::var("PATH")` を split して各バイナリ名を `which::which` でチェックするのと、`<project>/.claude/`、`<home>/.codex/` 等のディレクトリ存在チェックを並列で行います。7 エージェントだけなので `futures::future::join_all` で同時実行しても数十 ms 以内で終わるはずです。

`dialoguer::MultiSelect` は TTY を要求します。非 TTY 環境（CI）では `--non-interactive` 必須にし、指定なしで non-TTY を検出したらエラー終了します。

マージロジックは agent ごとに実装が違います:

- **Claude Code**: `.claude/settings.json` は JSON。`hooks.PostToolUse` / `hooks.Stop` 配列に kizu entry を push。既存 entry の `command` が `kizu hook-` で始まるものは skip（再 install 時のべき等）
- **Cursor**: `.cursor/hooks.json` は `{ version: 1, hooks: { afterFileEdit: [...], stop: [...] } }`。同様に push + dedupe
- **Codex**: `~/.codex/hooks.json` は Claude Code 相当。ただし PostToolUse entry は設置せず Stop のみ
- **Qwen Code**: `.qwen/settings.json` は Claude Code 相当
- **Cline**: `.clinerules/hooks/PostToolUse` に shell wrapper を配置。既存ファイルがあれば中に `kizu hook-post-tool --agent cline` 行を追加、無ければ新規作成
- **Windsurf**: `.windsurf/hooks.json` (Cascade Hooks 最新形式)。実装前に `docs.windsurf.com/windsurf/cascade/hooks` で event 名を再確認
- **Gemini CLI**: 何もしない、使用方法の stdout 出力のみ

`AgentInstaller` trait を定義して、各 installer を実装クラスとして持たせます。`run_init` はこの trait オブジェクトの Vec を順に呼ぶだけのシンプルなドライバに保ちます。

### M8（`kizu teardown`）

install の逆操作。`AgentInstaller::uninstall` で各設定ファイルから kizu 由来 entry を削除します。空になった `hooks.PostToolUse` 配列や `hooks` オブジェクトは削除し、元が空だった key は書き戻さない（完全に install 前の状態に戻す）。

## Idempotence and Recovery

- **M1（背景色移行）**: 完全に内部リファクタ。複数回実行されても影響なし。元に戻す場合は `git revert` で十分
- **M2-M4（scar 実装）**: `insert_scar` は同一 scar が直上にあれば no-op。`x` (revert) は複数回実行可能だが 2 回目は「既に revert 済み」なので no-op
- **M5（hook サブコマンド）**: 各 hook は純粋関数的に動作し、dedupe state `<state_dir>/scars/<session_id>.json` で重複通知を防ぐ。dedupe ファイルを削除すれば state は初期化される。event log は `<state_dir>/events/` で TTL 24 時間 + 上限 1000 エントリで自動 prune され、放置しても無制限成長しない
- **M6-M7（`kizu init`）**: 再実行しても既存 entry を `command` プレフィックスで判定して skip するため、重複 hook は入らない
- **M8（`kizu teardown`）**: 複数回実行しても初回以降は no-op。既に外されていればエラーにならない
- **失敗時のリカバリ**:
  - `init` 中のファイル書き込み失敗: tempfile → rename の atomic write なので元ファイルは破壊されない
  - `teardown` 中の書き込み失敗: 同様に atomic、失敗したファイルは原状のまま
  - hook サブコマンドが panic した場合: stdin JSON が malformed でも panic せず、`serde_json::Error` を stderr に出して exit 1 で終わる
- **環境のクリーンアップ**: `/tmp/kizu-scars/*.json` と `/tmp/kizu-events/*.json` は teardown では消さない（ユーザー判断）。完全クリーンアップコマンドが必要なら v0.3 で検討

### event log のファイルフォーマット（sanitize 済み）

保存先: `<state_dir>/events/<unix_ms>-<tool_name>.json`（`<state_dir>` は `$XDG_STATE_HOME/kizu/` 等）。ディレクトリは `0700`、ファイルは `0600`。

**生の stdin JSON ではなく、sanitize された `SanitizedEvent` のみを書き出す**:

```json
{
  "session_id": "01H...",
  "hook_event_name": "PostToolUse",
  "tool_name": "Edit",
  "file_paths": ["/path/to/foo.rs"],
  "cwd": "/path/to/repo",
  "timestamp_ms": 1745000000000
}
```

**削除するフィールド**:

- `tool_input.content` / `tool_input.new_string` / `tool_input.old_string` — edit したファイル本文
- `tool_response.output` / `tool_response.stdout` / `tool_response.stderr` — エージェントが観測した実行結果
- `prompt` / `user_prompt` / `assistant_message` — 対話ログ

**保持するフィールド**:

- `tool_input.file_path` / `tool_input.path` のような path metadata のみ
- agent 固有フィールドを `NormalizedHookInput` に正規化した結果

**retention**: TTL 24 時間、上限 1000 エントリ。`hook-log-event` 実行時に `prune_event_log` を呼ぶ。`KIZU_EVENT_TTL_SECS` 環境変数で TTL を上書き可能。
