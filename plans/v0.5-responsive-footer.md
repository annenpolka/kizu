# v0.5: フッターを幅に応じて縮退する responsive status bar にする

この ExecPlan はリビングドキュメントです。`Progress`、`Surprises & Discoveries`、`Decision Log`、`Outcomes & Retrospective` の各セクションは作業の進行に合わせて随時更新してください。

## Purpose / Big Picture

kizu のフッターは、現在のモード、対象ファイル、セッション全体の増減、wrap や行番号などのトグル状態、picker などのキー操作ヒント、watcher や git diff の異常を 1 行に表示している。画面が広いと便利だが、狭い pane や長いファイルパスでは右端から無条件に切れて、`w wrap`、`# nums`、`s picker` のような操作ヒントや異常表示が見えなくなる。

この変更では、フッターを **幅に応じて full / compact / minimal の候補から選ぶ responsive status bar** にする。さらに、キーマップ一覧はフッターから外して `?` の help overlay に移す。ユーザーは狭いターミナルでも、現在モード、重要な状態、wrap と `nums on` / `nums off` の状態を見失わない。キー操作を確認したい時だけ `?` を押せば、config remap 後のキー一覧を overlay で確認できる。

動作確認は、`cargo test responsive_footer` で狭幅のユニットテストが通ること、`cargo test help_overlay` で overlay の開閉と remap 表示が通ること、`cargo test footer` で既存フッター回帰が通ること、手元の TUI を狭い pane で開いて `[manual]`、`wrap`、`nums on/off` が右端で雑に切れず、`?` で help overlay が開くことを確認して行う。

## Outcomes & Retrospective

実装は `src/ui.rs` の footer 描画と help overlay 描画、`src/app.rs` の小さな overlay 状態追加に収まった。フッターは full / compact / minimal の 3 候補を作り、`area.width` に収まる最初の候補を選ぶようになった。狭い normal footer では長い path を捨て、session と `center` / `wrap` / `nums on` または `nums off` の状態を残す。file view では長い path より `Esc` の戻るヒントを優先する。広い footer でも keymap の列挙は避け、状態と `? help` に寄せた。

追加で、`?` help overlay が実装された。normal diff view と file view で `?` を押すと開き、Esc / `?` / `q` で閉じる。overlay が開いている間は `s` などの通常 action key を shadow するので、help を読んでいるだけで picker や scar が誤って動くことはない。overlay 内の keymap は `app.config.keys` の remap を反映する。

残課題は手動 TUI 確認である。`just ci` は通っており、release build と e2e まで確認済みである。

## Context and Orientation

kizu は Rust 製の terminal user interface、つまり端末上に状態を描くアプリケーションである。描画には `ratatui` を使う。`ratatui::text::Span` は色や太字を持つ短い文字列で、`Line` は複数の `Span` を横に並べた 1 行である。現在のフッターは `src/ui.rs` の `render_footer(frame, area, app)` が作る。

関連ファイルは以下である。

`src/ui.rs` は描画層で、`render_footer`、`render_help_overlay`、`pad_or_truncate_display`、テスト用の `render_to_string` がある。今回の主な編集対象である。実装後の `render_footer` は、mode span の後ろに branch ごとの span を順に push する直接方式ではなく、full / compact / minimal の候補を作って幅に収まるものを選ぶ。`render_help_overlay` は `?` で開く 2 カラムの keymap popup である。

`src/config.rs` はユーザー設定を表す。`KeyConfig` には `picker`、`cursor_placement`、`wrap_toggle`、`line_numbers_toggle` などがあり、入力処理はこれを使っている。footer も同じ値を使う必要がある。

`src/app.rs` は状態機械である。`handle_normal_key` と `handle_file_view_key` は `app.config.keys` を読んで実際のキー入力を処理する。今回の変更では `App.help_overlay: bool` を追加し、`handle_key` の最上位で help overlay が開いている時だけ `handle_help_key` に dispatch する。

`plans/v0.5-line-numbers.md` は行番号モードの既存 ExecPlan である。この計画では `# nums` を footer に出すことを必須にしていた。今回のユーザー判断により、footer は `nums` という状態表示を残し、操作キー `#` は help overlay に移す。

`plans/v0.5-newline-marker.md` は別件の EOF 改行なしマーカー計画である。今回の作業では触らない。

この計画で使う用語を定義する。**full 表示** は広い画面向けの説明付き表示で、例は `center │ wrap │ nums on │ ? help` である。**compact 表示** は中幅向けの短縮表示で、例は `center wrap nums off ? help` である。**minimal 表示** は狭い画面向けの状態だけを残す表示で、例は `wrap nums on` である。**help overlay** は `?` で開く一時 popup で、操作キー一覧を表示する。**truncate** は文字列を端末表示幅に合わせて短くし、収まる場合は末尾に `…` を付けることである。

## Plan of Work

最初に `src/ui.rs` と `src/app.rs` の tests module に Red テストを追加する。normal diff view の fixture では長いファイルパスを持つ `FileDiff` を作り、幅 64 程度で描画した footer が `[manual]`、`wrap`、`nums` を含み、`w wrap` や `# nums` のような keymap 表示を含まないことを確認する。file view の fixture では長い path と `lines.len()` を持つ `FileViewState` を作り、幅 56 程度でも `Esc` または `back` 相当、wrap 状態、nums 状態が残ることを確認する。App 側では `?` が help overlay を開き、Esc / `?` で閉じ、overlay 中は `s` などの通常キーを shadow することを確認する。UI 側では `app.config.keys.wrap_toggle = 'W'`、`line_numbers_toggle = 'L'`、`picker = 'p'` などに変え、help overlay が remap 後のキーを表示することを確認する。

次に `src/ui.rs` に footer 専用 helper を追加する。`spans_display_width(&[Span<'static>]) -> usize` は各 span の `content` を `unicode_width::UnicodeWidthStr` で測る。`choose_footer_variant(candidates, width)` は full、compact、minimal、mode-only の順に候補を受け取り、`width` に収まる最初の候補を返す。どの候補も収まらない極小幅では mode-only を返し、最終的な右端切れを ratatui に任せる。これは幅 10 未満の端末では完全な保証ができないためである。

その後、`render_footer` を「候補を作って選ぶ」形へ移す。normal branch の compact 候補は、path を `truncate_display` で短くし、session を `+N/-M Kf` に短縮し、状態を `center wrap nums ? help` のように詰める。minimal 候補は path を落とし、`[manual] +N/-M Kf HEAD* center wrap nums` のように状態の骨格だけを残す。

file view branch も同じ考え方で候補を作る。full では `wrap │ nums │ path [cursor/total] │ Enter/Esc back` を保つ。compact では `center wrap nums path cursor/total Esc back` にする。minimal では path を落とし、`[file view] cursor/total wrap nums Esc` だけを残す。

picker、search、scar comment、revert confirm の branch は現在の文言を full として保ち、compact と minimal では `Enter` / `Esc` や `(y/N)` など実行中モードで必須のキーを優先する。watcher、input、last_error の diagnostic は full では既存どおり message を表示し、compact / minimal では tag を優先して message を短くする。wide 幅の既存テストが期待する error text は維持する。

最後に `render_help_overlay` を追加する。`App.help_overlay` が true の時、`render` の最後で `render_help_overlay` を呼び、main/footer の上に Clear + bordered popup を重ねる。内容は 2 カラムで、左に Navigation / Review、右に Views / Search / Other を置く。各 action key は `app.config.keys` から `key_label` で表示する。

最後に既存テストを回し、必要なら期待値を「全幅では従来どおり、狭幅では縮退どおり」に調整する。`cargo fmt` は `just rust` の中で走るが、Red / Green の短い確認では個別 `cargo test` を使う。

## Idempotence and Recovery

この作業は `src/ui.rs`、`src/app.rs`、`plans/v0.5-responsive-footer.md` の追加・編集に限定する。テスト追加と helper 追加は何度実行しても同じ結果になる。`cargo test responsive_footer` と `cargo test help_overlay` は失敗しても作業ツリーを変更しないので、Red の確認は安全である。

## Interfaces and Dependencies

新規外部依存は追加しない。既に `unicode-width` が使われているため、footer の幅計測にも同じ crate を使う。

`src/ui.rs` に内部 helper を追加する。最終的に以下の関数が存在する。

    fn spans_display_width(spans: &[Span<'static>]) -> usize;
    fn truncate_display(s: &str, max_width: usize) -> String;
    fn choose_footer_variant(candidates: Vec<Vec<Span<'static>>>, width: u16) -> Vec<Span<'static>>;
    fn key_label(ch: char) -> String;
    fn render_help_overlay(frame: &mut Frame<'_>, area: Rect, app: &App);

`truncate_display` は padding をしない。既存 `pad_or_truncate_display` は表示幅を target ちょうどに揃える helper なので、footer の path や error message には padding なしの別 helper を使う。

`key_label` は `char` を表示文字列へ変換する。通常の 1 文字キーはそのまま表示し、space は `Space` と表示する。help overlay では `seen` が space なので、この helper で `Space` と表示する。

`src/app.rs` には以下の状態と handler を追加する。

    pub help_overlay: bool;
    fn handle_help_key(&mut self, key: KeyEvent);

`help_overlay` は起動時 false。normal diff view と file view で `?` を押すと true になり、true の間は `handle_key` の最上位で `handle_help_key` が入力を処理する。Esc / `?` / `q` は overlay を閉じ、Ctrl-C は従来どおり quit を許可する。

`src/config.rs` の public interface は変更しない。設定ファイル形式も変更しない。help overlay は既存 `KeyConfig` を読むだけである。
