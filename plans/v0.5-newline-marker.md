# v0.5: 改行マーカーの意味を反転する (常時 `¶` → EOF 改行なしの警告のみ)

この ExecPlan はリビングドキュメントです。`Progress`、`Surprises & Discoveries`、`Decision Log`、`Outcomes & Retrospective` の各セクションは作業の進行に合わせて随時更新してください。

## Purpose / Big Picture

kizu の diff view wrap モードは現在、長い diff 行が折り返された最終 visual row に `¶` (pilcrow) マーカーを右端に描画する。意図は「wrap 境界 (物理的な折り返し) と論理行末 (本物の改行) を区別する」こと。しかし実運用では以下の 3 点で視覚ノイズの割に情報量が少ない:

1. ほぼ全ての DiffLine は `has_trailing_newline = true` (= 改行あり) なので、マーカーは実質「全行の右端に付く常駐表示」になっている。区別としての価値が薄い
2. コード内の `\n` リテラル (`"[line_numbers]\nenabled = true\n"` 等) と `¶` が近接し、エスケープ表現の仲間に見えて読解を阻害する
3. v0.5 で行番号ガターが入った結果、wrap 継続行は **ガターが blank** になるので「wrap 境界」はそれで判別可能。`¶` 側が担うべき情報はガターに移譲されている

v0.5 では意味を **反転** する:

- **通常の論理行末 (has_trailing_newline = true) は無印**
- **EOF で改行なし (`has_trailing_newline = false`) の時だけ末尾に `∅` を描画**
- スタイルは `Color::Yellow + Modifier::BOLD` の警告色 (DIM なし)。背景色 (bg_added / bg_deleted) は継承して diff のコンテキストを保つ
- 適用範囲は **diff view nowrap / diff view wrap / file view nowrap / file view wrap の 4 経路すべて**。EOF 改行なしは wrap 有無と独立した情報なので、wrap のみに限定していた現状を拡張する

ユーザ観点の成果:

- 通常コードは画面に余計な記号が出ず、body が本来の形で読める
- ファイル末尾に改行がないレアケース (コードレビュー上は重要な情報) だけ Yellow bold で注意を引く
- git が `\ No newline at end of file` で表す情報と対応が取れる

動作確認:

1. 適当なリポジトリで `cargo run --release` 起動
2. 通常コード diff (全行に改行あり) を開く → **wrap ON/OFF 両方で `∅` / `¶` が出ない** ことを確認
3. `printf 'foo' > x.txt` のような **末尾改行なしのファイルを untracked** として作る → diff に `∅` が Yellow bold で表示される
4. 同じファイルを Enter で file view → 最終行の末尾に `∅` が付くことを確認
5. `echo foo >> x.txt` で改行を追加 → `∅` が消える

## Decision Log

- Decision: マーカーの意味を反転し、`has_trailing_newline = false` の時だけ描画する
  Rationale: 現状は 99% の行で `true` なので `¶` がほぼ常駐し情報量がない。`false` は git の `\ No newline at end of file` に対応する稀だが価値のある情報 → 出現頻度が低いものを強調した方がシグナル / ノイズ比が改善する。ユーザと対話済み。
  Date/Author: 2026-04-21 / ユーザ + Claude

- Decision: 記号は `∅` (U+2205 EMPTY SET)
  Rationale: 「欠落」「no-newline」の意味に直感的に繋がる。`¶` は論理段落の意味で、EOF 改行なしの警告には意味的にそぐわない。候補 `⎵` (BOTTOM SQUARE BRACKET) は文脈不明、`ø` / `⌀` も意味が弱い。`∅` は Unicode ブロックとしても「empty set」で一貫している。ratatui で幅 1 cell、多くの terminal フォントで安定描画。
  Date/Author: 2026-04-21 / Claude

- Decision: スタイルは `Color::Yellow + Modifier::BOLD` (DIM なし)
  Rationale: 現状の `¶` は `DarkGray + DIM` で常駐表示を緩和する目的だったが、新仕様では「稀なので注意を引きたい」が逆転した要件になる。Yellow + Bold は既存の cursor bar / selected marker と同系統の警告色で、ユーザの視覚言語に合致する。背景色 (bg_added / bg_deleted) は継承して diff コンテキスト (+/-) を保つ。
  Date/Author: 2026-04-21 / Claude

- Decision: 適用範囲を 4 経路 (diff view nowrap / wrap / file view nowrap / wrap) すべてに揃える
  Rationale: 現状は diff view wrap のみ。EOF 改行なしは wrap 有無と独立した情報なので、nowrap でも file view でも出すのが整合的。コードベースの `has_trailing_newline` は 4 経路すべてで利用可能。
  Date/Author: 2026-04-21 / Claude

- Decision: t-wada 流 TDD を遵守
  Rationale: 既存 `wrap_mode_renders_newline_marker_*` 系テストが旧仕様を pin しているので、Red (新仕様のテストを追加して旧実装で落ちる確認) → 既存テストの書き換え → Green の順で意味論転換を安全に進める。
  Date/Author: 2026-04-21 / Claude

### 専門用語

- **`¶`** (U+00B6 PILCROW SIGN): 現状の改行マーカー記号。段落記号として欧文組版に使われる
- **`∅`** (U+2205 EMPTY SET): v0.5 で採用する新記号。空集合 / 欠落を意味する
- **`has_trailing_newline`**: DiffLine が末尾に LF を持つかどうか。git は `\ No newline at end of file` で明示する場合のみ `false`
- **visual row**: wrap で 1 論理行が複数行に分かれた時の 1 行分
- **marker_reserve**: body_width の pad 計算時に「末尾にマーカー 1 cell を空けておく」量

### 既存テストの pin 具合 (src/ui.rs)

- `wrap_mode_renders_newline_marker_and_wraps_long_line` (line 2948 付近): 旧仕様「`has_trailing_newline = true` の時に `¶` が描画される」を pin → **書き換え必須**

### 2. Marker style ヘルパーを追加

```rust
fn eof_no_newline_span(bg: Option<Color>) -> Span<'static> {
    let mut style = Style::default()
        .fg(Color::Yellow)
        .add_modifier(Modifier::BOLD);
    if let Some(bg) = bg {
        style = style.bg(bg);
    }
    Span::styled("∅", style)
}
```

### 3. `render_diff_line` (nowrap) にも EOF マーカーを追加

現状 nowrap は `has_trailing_newline` を無視している。新仕様では nowrap でも `false` なら `∅` を描画する。pad 計算も追従。

### 4. `render_file_view_line` / `render_file_view_line_wrapped` にも追加

file view は現状マーカー完全無視。新仕様で `has_trailing_newline` 相当が必要だが、FileViewState は `lines: Vec<String>` を保持するだけで per-line の改行情報を持っていない。実装戦略:

- file view の lines は `String::split('\n')` で作られている (`open_file_view` 内)。**最終行が元ファイルで改行終端だったか否かだけ** `FileViewState.last_line_has_trailing_newline: bool` で持つ
- render_file_view で `line_idx == fv.lines.len() - 1 && !fv.last_line_has_trailing_newline` の時だけ `∅` を出す
- 中間行は全て改行あり扱い (split の性質から)

### 5. 既存テストの書き換え + 新規 Red テスト追加

- 既存 `wrap_mode_renders_newline_marker_and_wraps_long_line` を「**has_trailing_newline = true の時に `¶` も `∅` も出ない**」に反転
- 既存 `wrap_mode_omits_newline_marker_when_diff_line_has_no_terminal_newline` を「**has_trailing_newline = false の時に `∅` が出る**」に反転してリネーム (`wrap_mode_renders_eof_marker_when_no_terminal_newline`)
- 既存 `nowrap_mode_has_no_newline_marker` を「**nowrap + has_trailing_newline=true は無印**」に限定しつつ「**nowrap + has_trailing_newline=false は `∅` 出る**」を新規 pin
- 新規: `render_scroll_marks_eof_no_newline_in_yellow` (diff view 両モード)
- 新規: `render_file_view_marks_eof_no_newline_on_last_line`

### 6. `render_scroll` のコメントと body_width 式を更新

`wrap_body_width = area.width - 6` の 6 cell は「5 bar + 1 marker」。新仕様では「markerは稀にしか出ない」ため固定予約は微妙。ただし幅の予測不可能性を避けるため 1 cell は保持し、コメントを「5 for bar, 1 for optional EOF-no-newline marker」に更新。

### 7. docs/SPEC.md 追記

v0.5 節の行番号表示モードに続いて、改行マーカーの項目を追加:

- 通常の論理行末: 無印
- EOF 改行なし: `∅` を Yellow bold で描画 (git の `\ No newline at end of file` 対応)
- diff view / file view / wrap / nowrap すべてで共通
