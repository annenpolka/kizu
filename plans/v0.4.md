この ExecPlan はリビングドキュメントです。`Progress`、`Surprises & Discoveries`、`Decision Log`、`Outcomes & Retrospective` の各セクションは作業の進行に合わせて随時更新してください。

## Purpose / Big Picture

kizu の「既読 (seen) マーク」は現状 hunk header の左に `•` を付ける装飾のみで、差分本体 (DiffLine) はそのまま表示され続ける。AI コーディングエージェントと並走していると既読化済みの hunk も視界に残り続け、「まだ見ていない差分はどこか」をスクロールして目で探す必要がある。

v0.4 ではこの挙動を次のように変更する:

- 既読マーク済み hunk の **DiffLine 行を layout から除外** し、hunk header だけ残す (= 折りたたみ)
- 既読を付けた瞬間の hunk 内容から **1 文字でも変化したら自動的に既読解除** (= 自動展開)。変化検知は hunk 内容の fingerprint (u64 ハッシュ) 一致で判定する
- 既存のキーバインド (Space) とトグル動作、セッション限りの永続性はそのまま

ユーザ観点の成果:

- 一度レビューして Space で既読化した hunk は差分本体が消え、hunk header だけが残って視界がすっきりする
- AI が再度その hunk を編集すると (Added/Deleted/Context のどれかが 1 文字でも変わると) 自動的に展開されて再チェックを促せる
- 既読解除 (Space で再度トグル) でも展開できる

動作確認:

1. 適当なリポジトリで kizu を起動し、hunk が 1 つ以上ある状態を作る
2. カーソルを hunk header または hunk 内の DiffLine に置いて Space を押す
3. 当該 hunk の `-`/`+`/context 行がすべて非表示になり、hunk header だけが残ることを確認
4. 同じファイルの hunk 範囲 (baseline 側の `old_start` を動かさない範囲) を書き換える
5. kizu がリアルタイムに再計算し、当該 hunk が自動で展開される (DiffLine が再表示される) ことを確認
6. 再度 Space で手動折りたたみ → Space でもう一度押して手動展開できることを確認

## Decision Log

- Decision: 変化検知は「hunk 内容 fingerprint」方式 (B案) を採用する
  Rationale: kizu は AI コーディングエージェント並走が主用途。「一度レビューしたブロックを AI がまた触ったら再チェックしたい」という期待に合う。A 案 (pre-image anchor `old_start` のみ) では、同じ hunk 内で AI が追加編集しても既読のまま差分が隠れ続けてしまう。ユーザとの対話で B 案採用を合意。
  Date/Author: 2026-04-20 / ユーザ + Claude (Opus 4.7)

- Decision: fingerprint は `lines` 全体 (kind + content) でハッシュする
  Rationale: post-image (Added) の編集も pre-image (Deleted/Context) の動きも同等に検知したい。Added のみに絞ると、上流の行挿入で context が動いただけでは自動展開されず意図に反する。u64 一発のハッシュなので衝突は実質無視できる。
  Date/Author: 2026-04-20 / Claude

- Decision: `seen_hunks` のコレクションは `BTreeSet` → `BTreeMap` に変える (HashMap ではなく)
  Rationale: 既存も BTreeSet で決定論的な iteration/serialization を前提としている (セッションファイルに直接書いてはいないが、将来的な debug dump/持ち回しでの再現性)。`BTreeMap<K, V>` への格上げで既存の使用箇所の API 連続性を保ちつつ、value に fingerprint を持たせる。
  Date/Author: 2026-04-20 / Claude

- Decision: seen = 常に折りたたみ、展開状態の seen は存在させない
  Rationale: ユーザの要望は「既読で折りたたむ」。既読だが展開、のような中間状態を UI に持ち込むと状態空間が倍になりキー操作の意味も曖昧になる。「開きたければ未読に戻す」のほうが単純。`▸` のみ使い `▾` は未使用。
  Date/Author: 2026-04-20 / Claude

- Decision: t-wada 流 TDD を遵守する
  Rationale: seen 周辺は既存テスト (`space_toggles_seen_mark_on_current_hunk`, `seen_mark_persists_across_a_recompute_that_preserves_hunk_old_start`, file-header noop など) で pin 済み。fingerprint 方式への切り替えは後者のテスト意味論を変えるので、先に「新しい挙動を示す Red テスト」を書き、旧実装で期待通り失敗することを確認してから実装に入る。
  Date/Author: 2026-04-20 / Claude

### 専門用語

- **hunk**: `git diff` の `@@ -OLD_START,OLD_LEN +NEW_START,NEW_LEN @@` で区切られる一塊の変更ブロック。baseline (HEAD) 側の起点行 `OLD_START` が kizu の seen key の一部
- **old_start / new_start**: それぞれ baseline 側・worktree 側の 1-indexed 起点行番号
- **pre-image anchor**: ここでは `(path, old_start)` のペア。baseline 側で同じ位置を指し続けるための指標
- **fingerprint**: hunk の内容から計算する u64 ハッシュ。v0.4 で新規導入。既読時点の hunk 内容を記憶し、再計算後の hunk 内容と一致するかで「変化したか」を判定する
- **RowKind**: `rebuild_layout` が生成する表示行の種類。`FileHeader` / `HunkHeader { file_idx, hunk_idx }` / `DiffLine { file_idx, hunk_idx, line_idx }` / `BinaryNotice` / `Spacer` の 5 種 (src/app.rs:762)
- **ScrollLayout**: `rebuild_layout` が返す中間表現。`rows: Vec<RowKind>` を中心に `hunk_starts`, `file_first_hunk`, `file_of_row`, 変更走 (change run) 情報などを保持 (src/app.rs:584 付近)

### 既存テストの pin 具合 (src/app.rs の該当箇所)

- `space_toggles_seen_mark_on_current_hunk` (line 5734-): Space → insert → Space → remove を確認。**API 変更後も同じ意味で通す必要あり**
- `seen_mark_persists_across_a_recompute_that_preserves_hunk_old_start` (line 5787-): recompute 後も seen 保持。**fingerprint 方式で意味が変わる** → lines も変わらない前提なら保持。この条件を明示する形で書き換える
- `file-header Space must not add anything to seen_hunks` 系 (line 5781 付近): file header 上で Space は no-op。変更なし

### 3. Green: `rebuild_layout` で seen hunk の DiffLine をスキップ

## Idempotence and Recovery

- **TDD サイクルは冪等**: Red テストは毎回同じメッセージで落ち、Green 実装後は何度実行しても緑
- **作業中断時**: `Progress` のチェックボックスを最新状態にし、次回再開時はそこから続行
- **ロールバック**: `git reset --hard HEAD` で v0.3.3 時点に戻せる (destructive なのでユーザ確認必須)
- **部分的リカバリ**: BTreeMap 化だけ終わって fingerprint 実装が未完の場合でも、`is_hunk_seen` が `current_fp` を無視して key 存在だけ見る実装にしておけば A 案相当で動作するので、PR を 2 分割できる
