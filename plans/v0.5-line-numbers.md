# v0.5: 行数表示モード (line number gutter) を diff view と file view に追加する

この ExecPlan はリビングドキュメントです。`Progress`、`Surprises & Discoveries`、`Decision Log`、`Outcomes & Retrospective` の各セクションは作業の進行に合わせて随時更新してください。

## Purpose / Big Picture

kizu は AI コーディングエージェントの出した変更をリアルタイムに覗き込むツールだが、現状は diff 本文にも file view にも行番号が一切表示されない。エージェントが `src/foo.rs の 142 行目` と返してきた時に、kizu の画面内で「その 142 行目がどこか」を目視で特定する手段が無く、フッタの `[cursor/total]` かエディタ (`e`) を別途起動するしかない。

v0.5 では、**左ガターに行番号を常設表示できる「行数表示モード」** を追加する:

- **diff view**: 各 DiffLine の左側に `old | new` 2 列の行番号を表示する
  - Context: `old new` 両方表示 (`  10  10`)
  - Added: new のみ (`       11`)
  - Deleted: old のみ (`  11     `)
- **file view**: 各行に 1-indexed の行番号を 1 列表示する
- **Stream mode では行番号を表示しない**: Stream mode の FileDiff は PostToolUse イベントから合成した per-operation diff で、`old_start/new_start` は `anchor_base` に基づく合成値。これを「実ファイル行番号」として表示すると誤認リスクが高いので、**`App.view_mode == ViewMode::Stream` のときは `show_line_numbers` が ON でもレンダラで行番号 gutter を抑制する**
- **wrap モード**: 1 論理行が複数の visual row に分かれた時、**最初の visual row だけに行番号を表示し、続き行は空白**にする (行番号の重複表示を避ける)
- **トグル**: 既定は OFF (非表示)。キー `#` (既定) と `~/.config/kizu/config.toml` の `keys.line_numbers_toggle` でリマップ可能。diff view でも file view でも同じキーで切り替わる
- 設定で初期値を指定可能: `[line_numbers].enabled = true`

ユーザ観点の成果:

- エージェントが `142 行目を修正した` と言ってきたら、kizu の画面上で即座に該当行を目視できる
- scar を挿入しようとしている行が「ソース全体のどの行か」を離れた場所からでも一望できる
- `#` を押せば行番号が出て、もう一度押せば消える (画面幅を取り戻せる)
- 長い行を wrap 表示している時も、行番号列が折り返し続き行で空白になるので「論理行の区切り」が視覚的に分かる

動作確認:

1. 適当な Rust プロジェクトで `cargo run --release` で kizu を起動する
2. diff が 1 件以上表示された状態で `#` キーを押す → 左ガター (既存の 5 幅のカーソルバーのすぐ右) に `old new` 2 列の数字が現れる
3. もう一度 `#` を押す → 行番号が消える (画面幅が戻る)
4. Added 行 (緑背景) には new 側のみ、Deleted 行 (赤背景) には old 側のみの数字が出る
5. フッタに `#` `nums` (シアン、行番号 ON 時は bold) のヒントが出る
6. `w` で wrap モードに切り替え → 長い行が 2 行以上に折り返された時、先頭 visual row だけに番号が出て続き行は空白になる
7. Enter で file view に入る → ファイル全体の左に `  1`, `  2`, …と行番号が出る (`#` で toggle)
8. file view で wrap を ON にしても継続行が空白で pin される
9. Tab で Stream mode に切り替え → **行番号は出ない** (実ファイル行番号ではないため)。Tab でメイン diff に戻すと再び表示される
10. `~/.config/kizu/config.toml` に `[line_numbers]\nenabled = true` を書いて起動 → 初期状態で行番号が表示される
11. `[keys]\nline_numbers_toggle = "L"` を書く → `#` ではなく `L` で toggle されるようになる

## Decision Log

- Decision: diff view は `old | new` 2 列、file view は 1 列で表示する **(後に撤回、下記参照)**
  Rationale: diff は pre-image と post-image を両方見ているので片方だけだと「反対側の行番号が知りたい」が頻発する (delta / diff-so-fancy も 2 列)。file view は単一のスナップショットなので 1 列で十分。Added は new のみ、Deleted は old のみという非対称は「そもそもその行は反対側には存在しない」事実を視覚的に伝える。
  Date/Author: 2026-04-21 / ユーザ + Claude

- Decision: **2 列案を撤回。diff view も 1 列 (new 側のみ) に統一し、Deleted 行は gutter を空白にする**
  Rationale: 実機確認 (2026-04-21) で 2 つの問題が判明。
    (1) **Context 行で old == new が常なので「13 13」「14 14」と同じ数字が並び、視覚的に "二重表示" で bug に見える** (ユーザフィードバック「バグってる」「二重に表示されている」)。
    (2) より根本的に、**hunk 外の +N 変更があると、同じ論理位置の Deleted 行と Added 行で old / new 番号が大きくズレる**。例: ファイル上部で 10 行 Added されていると、Hunk #2 の Deleted L5 と Added L15 が隣接する。これは unified diff の本質的な性質だが、gutter に両方混ぜると「行番号列が連続してない」と感じて読みにくい。
  worktree 側 (new) のみ表示すれば gutter は常に単調増加で、ユーザが「今見ているファイルのどこ」を直感的に掴める。Deleted 行の old 側は「現在のファイルには存在しない行」なので空白が自然 (bat / delta の `--file-style=omit` 相当)。baseline 位置情報が必要な場合は `e` (editor) の外部ツールに委ねる。
  Date/Author: 2026-04-21 / ユーザ + Claude (実機確認に基づく UX 修正)

- Decision: 既定キーは `#` を採用し、既定状態は OFF にする
  Rationale: `l` は既存の `KeyCode::Char('l')` ナビゲーション (app.rs:2000) と衝突。`n` は `search_next`。`L` は未使用だが Shift が必要で左手操作が増える。`#` も Shift だが視覚的に「行番号マーカー」として直感的 (vim の `#` レジスタ、git の行番号表記など)。既定 OFF は現状の画面幅を保ったままで後方互換を保つため。
  Date/Author: 2026-04-21 / Claude

- Decision: wrap 時の続き行は行番号を空白で埋める (論理行番号を繰り返さない)
  Rationale: 同じ番号を何度も繰り返すと「別の行」に見える認知負荷が高い。空白にすると「ここは折り返しの続き」が一目で分かる。delta や bat が採用している表記に揃える。
  Date/Author: 2026-04-21 / Claude

- Decision: 行番号 gutter の幅は「可視 DiffLine の最大行番号の桁数」で動的に決める (seen hunk は除外)
  Rationale: Codex review §Important-1 の指摘。全 hunk から max を取ると、折りたたみ済みの seen hunk (v0.4 で非表示) によって gutter 幅だけが広がる現象が起きる。`rebuild_layout` が layout を組みながら可視 DiffLine のみを走査して max を出せば、画面に実際に出る番号に gutter が揃う。最小 2 桁下限は維持。
  Date/Author: 2026-04-21 / Claude + Codex adversarial review

- Decision: **Stream mode では `show_line_numbers` に関わらず行番号を表示しない**
  Rationale: Codex review §Critical-2 の指摘。Stream mode の FileDiff は PostToolUse イベントから合成された per-operation diff で、`old_start/new_start` は `anchor_base` に基づく合成値であり実ファイル行番号とは一致しない。これを「行番号」として表示すると誤認リスクが非常に高い。無効化の判定は `App.view_mode == ViewMode::Stream` をレンダラ側で見る。footer には「行番号は stream では無効」という微弱なヒント (dim 色) を出す。
  Date/Author: 2026-04-21 / ユーザ確認 + Codex adversarial review

- Decision: `body_width` の算出は `render_scroll` / `render_file_view` に一箇所で集約し、レンダラ内部では縮めない
  Rationale: Codex review §Critical-1 の指摘。wrap 配置計算 (`FileViewVisualIndex::build`, `wrap_at_chars`) と描画本体が同じ `body_width` を前提にしていないと、`skip_visual` / `cursor_sub_row` / `¶` marker 位置がズレる。行番号 gutter 分を縮めた `body_width` を一度だけ算出し、VisualIndex と新レンダラの両方に同じ値を渡す。これは単一責任の強い契約変更なので Decision Log にも記録し、ADR-0015 への昇格可否を実装中に判断する。
  Date/Author: 2026-04-21 / Claude + Codex adversarial review

- Decision: `line_numbers_for` は `rebuild_layout` 時に先計算し、`ScrollLayout.diff_line_numbers: Vec<Option<(Option<usize>, Option<usize>)>>` として **`rows` と並列の 1 次元 Vec** で保持する (HashMap は使わない)
  Rationale: Codex review §Important-2 の指摘。単純 O(line_idx) 実装は大 hunk で render コストが跳ねる。2 巡目 §Important-1 の追加指摘で `Vec<Vec<_>>` vs `HashMap<(usize,usize), _>` の表記ゆれを指摘され、`build_layout` が安定順に走査 (src/app.rs:3844-3863) する前提を活かして **`rows` と同 index の並列 Vec** に統一。row index → `diff_line_numbers[i]` の直接 lookup は O(1) かつ cache friendly。`RowKind::DiffLine` 以外の row (HunkHeader / FileHeader / Spacer / BinaryNotice) の位置には `None` が入る。seen hunk の DiffLine はそもそも push されないので問題にならない。
  Date/Author: 2026-04-21 / Claude + Codex 2 巡目 adversarial review

- Decision: `body_width` の算出式と極小幅 fallback を具体化する
  Rationale: 2 巡目 Codex review §Critical-2 の指摘。1 巡目で「一箇所に集約」と言ったが、`saturating_sub` の位置と極小幅の挙動が未固定だった。契約は以下に確定する:
    - `effective_show_ln = app.show_line_numbers && app.view_mode != ViewMode::Stream`
    - `ln_gutter_width = if effective_show_ln { 2 + ow + 1 + nw + 2 } else { 0 }` (diff view)
    - `ln_gutter_width = if effective_show_ln { 1 + w + 1 } else { 0 }` (file view)
    - `body_width = (area.width as usize).saturating_sub(5 + ln_gutter_width).max(1)`
    - **極小幅 fallback**: `area.width < 5 + ln_gutter_width + 4` (= 本文 4 cell 未満) の場合、`effective_show_ln` を false に落とし、`ln_gutter_width = 0` で再計算する
    - `fv.last_body_width.set(body_width)` / `FileViewVisualIndex::build(&fv.lines, Some(body_width))` / `wrap_at_chars(content, body_width.max(1))` すべてに同じ値を渡す
  Date/Author: 2026-04-21 / Claude + Codex 2 巡目 adversarial review

- Decision: `toggle_line_numbers` と `ViewMode` 切替は `reflow_file_view_after_width_change()` を必ず呼ぶ
  Rationale: 2 巡目 Codex review §Important-2 の指摘。`rebuild_layout` だけでは file view の `FileViewVisualIndex` と scroll 位置が旧 body_width 前提のまま取り残される。既存の `reflow_file_view_after_wrap_toggle` (src/app.rs:3212-3225) と同じパターンで、body_width 変化時に VisualIndex を rebuild し、cursor_sub_row をクリアし、viewport を cursor 中心に再計算する `reflow_file_view_after_width_change()` を新設。`toggle_line_numbers` / `ViewMode` 切替 / (将来の) ウィンドウリサイズ の 3 箇所から呼ぶ。
  Date/Author: 2026-04-21 / Claude + Codex 2 巡目 adversarial review

- Decision: file view wrap + LN toggle の reflow タイミングズレは v0.5 では直さない (既知の制限)
  Rationale: Codex 3 巡目 review §Important-2 の仮説指摘。`reflow_file_view_after_width_change` は `fv.last_body_width.get()` で VisualIndex を組むが、その値は **前回 render の結果** なので LN toggle の瞬間 (= ON/OFF 切替の 1 フレーム) だけ body_width が古い値になる。ただし既存の `reflow_file_view_after_wrap_toggle` も同じ構造で、wrap flag 切替時に同じ 1 フレームのズレが発生する (v0.4 以前から存在)。v0.5 単独の regression ではなく共通の構造問題なので、本 PR ではスコープ外。根本修正には App 側で「現在の effective body_width」を area.width から自前算出する helper を用意して reflow 時に共有する必要がある → 将来の refactor で対応。
  Date/Author: 2026-04-21 / Claude + Codex 3 巡目 adversarial review

### 専門用語

- **hunk**: `git diff` の `@@ -OLD_START,OLD_LEN +NEW_START,NEW_LEN @@` で区切られる一塊の変更ブロック
- **old_start / new_start**: それぞれ baseline (HEAD) 側・worktree 側の 1-indexed 起点行番号
- **DiffLine**: ハンク内の 1 行を表す型。`kind` (Context/Added/Deleted) と `content` (末尾改行を除いた本文) を持つ
- **LineKind::Context**: 変更のない context 行。old/new 両方で同じ行番号を占める
- **LineKind::Added**: worktree 側にしか存在しない追加行。new 側にだけ行番号がある
- **LineKind::Deleted**: baseline 側にしか存在しない削除行。old 側にだけ行番号がある
- **body_width**: `render_*` 関数が文字列本文を描画できる幅 (= `area.width - gutter_width`)。現状すべて 5 固定 (`     ` / `  ▶  ` / `  ▎  ` の幅)
- **line-number gutter**: v0.5 で新設する、左の既存 gutter の右隣に挟まる行番号表示領域
- **visual row**: wrap モードで 1 論理行を body_width で折り返した結果の 1 行分。`line_idx = 10` の論理行が 3 visual row に分かれる場合、visual row インデックスは 0, 1, 2

### 2. 手動確認 (実機 TUI)

`cargo run --release` で kizu を起動し、以下を順に確認 (マトリクス: view × wrap × 行番号 × Stream):

- **diff view / wrap OFF**:
  - 起動直後、行番号は非表示
  - `#` を押す → 左ガター直後に `old | new` 2 列が現れる
  - Context 行には両方、Added 行には new のみ、Deleted 行には old のみ数字が出る
  - 数字は DarkGray + DIM で、コード本文とは色で明確に区別される
  - フッタに `#` `nums` (bold) が表示される
  - もう一度 `#` → 消え、フッタの `nums` も非 bold に
- **diff view / wrap ON**:
  - `w` で wrap ON、長い行 (80 cells 超) を探す
  - `#` で行番号 ON → 折り返された行の先頭 visual row にだけ行番号が出る。続き行は空白
  - `¶` marker (末尾改行) の位置が行番号 ON/OFF で動かない (Codex review §Critical-1 の配置一貫性)
  - カーソルの `▶` 位置も ON/OFF で同じ行を指す
- **file view / wrap OFF**:
  - Enter でファイルビュー → `#` で行番号 ON → ファイル全体の左に 1-indexed 連番が出る
  - 1 行ファイル、1000 行超ファイル両方で gutter 幅が正しく変わる
- **file view / wrap ON**:
  - 長い 1 行を含むファイルを開く → `w` + `#` で、先頭 visual row のみ番号、継続行は空白
  - Esc で diff view に戻る → 行番号モードは維持される
- **Stream mode** (Codex review §Critical-2):
  - Tab で Stream mode に切替
  - `#` を押しても行番号 gutter は出ない (画面幅も変わらない)
  - フッタには `#` `nums` が出るが dim 灰色で `(off)` 併記
  - Tab でメイン diff に戻すと、行番号 ON/OFF が再び有効になる
- **seen hunk (v0.4)** (Codex review §Important-1):
  - 大きい hunk (行番号 500 超) を Space で folded → `#` ON
  - 別の可視 hunk (行番号 100 以下) のみで gutter 幅が決まる (500 超まで広がらない)
  - folded hunk の DiffLine は当然出ない (行番号 gutter も出ない)
- **config**:
  - `~/.config/kizu/config.toml` に `[line_numbers]\nenabled = true` を書いて再起動 → 初期から行番号 ON
  - 同ファイルに `[keys]\nline_numbers_toggle = "L"` を書いて再起動 → `#` が効かず、Shift+l で toggle する
  - 衝突: `line_numbers_toggle = "a"` を書くと起動時の stderr 警告に出る (`kizu: warning: config key 'a' is bound to multiple actions: ask, line_numbers_toggle`)

### 3. 回帰確認

- scar 挿入 (`a` / `r` / `c`) が行番号モードで正しく動作 (scar は body 側の行に対して挿入、gutter には影響しない)
