# kizu v0.1 MVP — リアルタイム diff 監視 TUI の実装

この ExecPlan はリビングドキュメントです。`Progress` / `Surprises & Discoveries` / `Decision Log` / `Outcomes & Retrospective` の各セクションは作業進行に合わせて継続的に更新します。

## Purpose / Big Picture

このマイルストーンを完了すると、ユーザーは `cargo run` で kizu を起動したまま、別ペインで Claude Code 等のエージェントがファイルを書き換える様子を**リアルタイム**に観察できるようになります。具体的には:

- `cargo run` で kizu TUI が起動し、カレントディレクトリの git worktree を監視する
- 左ペインにセッション開始時 HEAD との差分がある変更ファイル一覧、右ペインに選択ファイルの unified diff、下部フッターにセッション統計が表示される
- 別ターミナルで `echo hello >> src/main.rs` のように書き換えると 300ms 以内に画面が更新される
- `j`/`k` でファイル選択、`f` でフォローモード切替、`q` で終了
- `R` (Shift+r) でセッションベースラインを現在の HEAD に即座にリセットする
- `git diff` がエラーを返した場合はフッタに赤い `×` とメッセージを出し、前回成功時の `files` を保ったまま運転を継続する
- 初回コミットが存在しない git リポでも空 tree SHA を baseline として正しく起動する（worktree 内ファイルがすべて追加扱いで表示される）
- バイナリファイルが変更された場合は左ペインのリストに `bin` として表示し、右ペインに `[binary file - diff suppressed]` を出す

加えて、tuistory + bun による e2e テスト 5 本（起動→q クリーン終了 / ファイル書き換え反応 / j/k+follow トグル / R リセット / added=緑・deleted=赤 色検証）が CI で自動実行され、これらの観察可能挙動を回帰から守ります。

scar 機能（`a`/`r`/`c`/`x`）、Claude Code フック、`--attach` 等は v0.2 以降のスコープとし、このマイルストーンには含めません。**純粋な観察ツールとして完結させる**ことがゴールです。

## Decision Log

実装中に行った設計判断は全てここに記録します。

- Decision: diff 計算はライブラリを使わず `git` CLI を shell out する
  Rationale: `--no-renames` 等の高度なフラグを再実装する手間を避け、git の挙動とのズレを生まない。詳細と代替案の却下理由は `docs/adr/0001-git-cli-shell-out.md` を参照。
  Date/Author: 2026-04-15 / Initial plan

- Decision: ファイル監視は `notify-debouncer-full 0.7` を使用する
  Rationale: worktree / `.git/HEAD` の分離デバウンス (300ms / 100ms) とバーストイベント集約を自前実装より安全に行える。詳細と代替案の却下理由は `docs/adr/0002-notify-debouncer-full.md` を参照。
  Date/Author: 2026-04-15 / Initial plan

- Decision: 非同期ランタイムとして tokio (`current_thread` flavor) を v0.1 から採用する
  Rationale: イベント駆動で `crossterm::event::EventStream` と watcher チャネルを `tokio::select!` で統合し、v0.2/v0.3 の hook / stream mode 拡張をシームレスにする。`compute_diff` 等の git shell out は `std::process::Command` のままとし、async の伝染を `app::run` の境界で止める。詳細は `docs/adr/0003-tokio-async-runtime.md` を参照。
  Date/Author: 2026-04-15 / Plan revision

- Decision: v0.1 では kizu 側に `.gitignore` フィルタを **持たない**。`git diff` 自体の `.gitignore` 尊重と watcher → app の coalescing で吸収する
  Rationale: Codex adversarial review で当初プランの `ignore` crate 採用が API 誤用 (`GitignoreBuilder::new` は何も読まない、ネスト `.gitignore` も `dir::Ignore` が `pub(crate)` で扱えない) と判明。フィルタを正しく実装するコストの方が、フィルタを持たないコスト (バーストで `compute_diff` が無駄に走る) より高い。coalescing を入れれば 50000 件のバーストが 1 回の `compute_diff` 呼び出しに吸収される。詳細は `docs/adr/0005-watcher-coalescing-no-ignore-filter.md`。
  Date/Author: 2026-04-15 / Plan revision (Codex review fix)

- Decision: git ディレクトリは `git rev-parse --absolute-git-dir` で解決し、`<root>/.git` をハードコードしない
  Rationale: linked worktree (`git worktree add` で作られたサブ worktree) では `.git` がファイルでありポインタ先の実体パスは別になる。当初プランの `<root>/.git/` プレフィックス除外と HEAD watcher は linked worktree で動かない。`src/git.rs::git_dir(root)` を新設して shell out で解決し、`App.git_dir` として保持する。
  Date/Author: 2026-04-15 / Plan revision (Codex review fix)

- Decision: watcher → app 境界に **drain ベースの coalescing** を入れる
  Rationale: `unbounded_channel` を読み出すたびに 1 件しか処理しないと、バーストイベント時にキューが伸び続けキー応答が遅延する。1 件受信したら `try_recv` ループで残りを全部吸収し、`Worktree` フラグが立っていれば `compute_diff` を 1 回だけ呼ぶ。`tokio::sync::Notify` や `watch::channel` への切り替えも検討したが、`Worktree`/`GitHead` の種別を残しつつ実装行数が最小になるのは drain パターン。詳細は ADR-0005。
  Date/Author: 2026-04-15 / Plan revision (Codex review fix)

- Decision: ci.yml の bun + e2e ステップは `tests/e2e/package.json` の存在ガードでスキップ可能にする
  Rationale: 当初プランは required check として bun ステップを入れたが、`tests/e2e/` の scaffold は M6 で land する。先に CI を入れると M1〜M5 を incremental に PR で land できなくなる (= 全 PR が即 fail)。`steps.e2e_scaffold.outputs.present == 'true'` でガードし、scaffold が存在しない間はステップ自体をスキップする。

  Rationale: CLAUDE.md の t-wada 流 TDD 方針。純粋関数は Red/Green/Refactor を最小サイクルで回せるため実装速度が上がる。
  Date/Author: 2026-04-15 / Initial plan

- Decision: ベースラインリセットは `R` (Shift+r) 1 打で即時実行する（`r` 2 連打ではない）
  Rationale: SPEC.md の当初仕様は「`r` 2 連打」だったが、実装すると `reset_pending: Option<Instant>` を `App` に持たせる必要があり、誤爆検出の時間しきい値を選ぶ根拠も乏しい。`R` 1 打なら確認状態が不要で、誤爆しても次の `R` で現在の HEAD に再固定するだけなので副作用が小さい。SPEC.md と CLAUDE.md も同時に更新済み。
  Date/Author: 2026-04-15 / Plan revision

- Decision: `git diff` がエラーを返した場合は `App.last_error: Option<String>` に記録してフッタ右端に赤の `×` + メッセージを表示し、`files` は前回成功時の状態を保持する
  Rationale: panic させると Claude Code 作業中に kizu が一瞬で落ちて作業が中断する。`files` を空にすると「本当に diff なし」と区別できず誤認識の元。フッタ表示 + 前回状態保持なら「一時的な達なり」を見て見ぬふりできて、次の watcher イベントで自動リトライされる。成功したら `last_error` をクリア。bootstrap 段階のエラー（`find_root` / 初回 `compute_diff`）は stderr に吐いて exit する（fail-fast）。
  Date/Author: 2026-04-15 / Plan revision

- Decision: 初回 commit がない git リポでは empty tree SHA (`4b825dc642cb6eb9a060e54bf8d69288fbee4904`) を baseline として使う
  Rationale: `git rev-parse HEAD` が失敗するケースを特殊化するとコードパスが分岐する。空 tree SHA を baseline にすれば「全 worktree 内ファイルが追加扱い」として diff が計算され、コードパスが 1 本になる。`git init` 直後のスクラッチ作業もそのまま観察できる。
  Date/Author: 2026-04-15 / Plan revision

- Decision: バイナリファイルは `BinaryDiff` バリアントとしてリストに表示し、右ペインは `[binary file - diff suppressed]` プレースホルダー
  Rationale: リストから除外するとエージェントが PNG を誤って上書きしたケースを見逃す。`FileDiff` に `content: DiffContent` enum を持たせ、`Text(Vec<Hunk>)` と `Binary` を分ける。`+N/-M` 表示は `bin` に置き換える。
  Date/Author: 2026-04-15 / Plan revision

- Decision: e2e テストに tuistory + bun を v0.1 から主力採用する
  Rationale: kizu の価値は visual diff observation そのものなので、実 pty + 色フィルタ + key イベント配送 + raw mode 復旧を全てカバーできる tuistory が最適。`tests/e2e/` を新設し、bun:test の inline snapshot で 5 本 (smoke / reactive / navigation / R reset / colors) を書く。CI では既存の単一ジョブに bun ステップを追加。詳細と代替案の却下理由は `docs/adr/0004-tuistory-e2e.md` を参照。
  Date/Author: 2026-04-15 / Plan revision

- Decision: ファイル選択は **パス追跡** で行う。`App.selected_path: Option<PathBuf>` を保持し、recompute_diff のたびに新 `files` から同じ path を探してインデックスを再計算する
  Rationale: インデックス追跡だと watcher イベントで他のファイルが追加/削除されたときセレクションが暗黙にずれる。ユーザーの「これを見ている」コンテキストを坊さない方が観察ツールとして自然。同じ path が見えなくなったら直前のインデックスに clamp してフォールバック。
  Date/Author: 2026-04-15 / Plan revision

- Decision: フォローモードのターゲットは `files` の中で **mtime が最新** のファイル（`std::fs::metadata().modified()`）
  Rationale: 「最新の変更ファイル」を機械的に決める一番自然な定義。watcher イベントから path を伝搬させる方法は ignore フィルタとの順番バグを誘発するので避ける。deleted ファイルは mtime 取得不能なので follow 対象外（mtime 取得失敗時は `UNIX_EPOCH` 扱いでソート末尾に行く）。
  Date/Author: 2026-04-15 / Plan revision

- Decision: ファイルリストは **mtime 降順** でソートする
  Rationale: 「いま何が動いているか」が一目で分かるレイアウト。follow target も常にリスト先頭に見える。パス追跡セレクションと組み合わせると、選択中のファイルがリスト上を動くことは許容するが、ユーザーは「path を見ている」のでカーソル位置のずれは気にならない。
  Date/Author: 2026-04-15 / Plan revision

- Decision: 1 ファイルの diff は **2000 行で truncate** し、末尾に `[+N more lines truncated]` を表示する
  Rationale: 10000 行の untracked / 5000 行の一括書き換えで syntect とラインベース描画が固まることを防ぐ。truncate は UI レンダリングレベルで行い、`FileDiff` 自体には全行入れる（将来 v0.1.1 で「展開キー」を実装するときの素地として）。untracked 内容は `std::fs::read_to_string` を 8KB で cap し、それ以上は `[+N more bytes from new file]` と表示する。
  Date/Author: 2026-04-15 / Plan revision

- Decision: 右ペイン in-file スクロールは **`Ctrl-d` / `Ctrl-u` (半ページ) + `g` / `G` (先頭/末尾)**、follow モード ON のときは recompute のたびに **最後の hunk へ自動スクロール**
  Rationale: lazygit / delta 慣習に近く、`j`/`k` はファイルリスト操作と切り分けられる。手動スクロールしたら follow を `manual` にダウングレードし、`f` で復帰する（ファイルレベルと同じセマンティクス）。「最後の hunk」とは `new_start` が最大の hunk。
  Date/Author: 2026-04-15 / Plan revision

- Decision: worktree が clean (差分ゼロ) のときは **両ペインに中央揃えで `No changes since baseline (HEAD: <short_sha>)`** を表示する
  Rationale: 「フリーズした」と「本当に何もない」を視覚的に区別する。e2e smoke test は `waitForText('No changes')` でこの状態を assert する。フッタは `[follow] -- | session: 0 files`。
  Date/Author: 2026-04-15 / Plan revision

## Context and Orientation

### 用語の定義

- **kizu**: 本プロジェクト。Rust 製の単一バイナリ TUI で、AI コーディングエージェント（Claude Code 等）と並走してリアルタイムに diff を監視する。
- **worktree**: git リポジトリのうち、実ファイルが展開されているディレクトリツリー。`.git` を含まないユーザーが編集するファイル群。
- **baseline SHA**: kizu セッション開始時の HEAD コミット SHA。diff はこの SHA と現在の worktree の差分として計算される。
- **unified diff**: `git diff` が出力する標準形式。`@@ -old_start,old_count +new_start,new_count @@` の hunk ヘッダと、` `（context）/`+`（added）/`-`（deleted）の行接頭辞で構成される。
- **hunk**: unified diff における連続した変更ブロック。old/new の開始行と行数を持つ。
- **フォローモード**: 最新の変更ファイルに自動的にセレクションをジャンプさせるモード。手動ナビゲーションで一時停止、`f` キーで再開。
- **scar**: ファイル内に直接書き込む `@review:` インラインコメント（v0.2 以降）。v0.1 では実装しない。
- **デバウンス**: 短時間に連続して届くイベントを 1 つにまとめる処理。notify crate はファイルシステムイベントをバーストで受信するため必須。

## Plan of Work

以下の 6 マイルストーンを順に実施する。各マイルストーンは独立してコンパイル・テスト可能な状態で終える。M1〜M5 が Rust 本体実装、M6 が tuistory + bun による e2e テスト整備。

### Milestone 1: `src/git.rs` — diff 計算層（純粋関数 + shell out）

**t-wada 流 TDD で進める**。まず `src/git.rs` 内に `#[cfg(test)] mod tests` を切り、純粋関数である unified diff パーサから Red → Green → Refactor のサイクルを回す。

1. **型を更新**: `FileDiff` の中身を `content: DiffContent` enum に持たせ、`enum DiffContent { Text(Vec<Hunk>), Binary }` とする。これによりバイナリ差分を構造的に表現できる。
2. **Red**: `parse_unified_diff` 関数のテストを書く。固定の unified diff 文字列（1 ファイル / 1 hunk / 1 行 added）を入力にして、期待される `Vec<FileDiff>` を assert する。まだ関数が存在しないのでコンパイルエラー、これも Red。
3. **Green**: 最小実装。入力行を走査し、`diff --git a/... b/...` ヘッダで `FileDiff` を開始、`@@ -... +... @@` で `Hunk` を開始、` `/`+`/`-` 接頭辞で `DiffLine` を push する。仮実装でベタ書きしてよい。
4. **Refactor**: 行の分類ロジックを `LineKind::from_prefix` のような関数に切り出す。
5. **三角測量**: 2 ファイル / 複数 hunk / modified+deleted 混在のテストを追加し、実装を一般化する。`FileStatus` は diff の `new file mode` / `deleted file mode` ヘッダから決定する。
6. **バイナリ差分の検出**: `Binary files a/foo and b/foo differ` 行を検出したら `DiffContent::Binary` バリアントの `FileDiff` を追加する。テストで固定文字列を入力して assert。
7. **`added` / `deleted` カウント**: `DiffLine::kind` を集計して `FileDiff.added` / `FileDiff.deleted` を埋めるテストを追加。バイナリは `added=0/deleted=0` で `bin` 表示扱いとする。
8. **shell out 層**: `find_root` は `git rev-parse --show-toplevel`、`head_sha` は `git rev-parse HEAD`、`compute_diff` は `git diff --no-renames <baseline> --` を `std::process::Command` で実行し、stdout を `parse_unified_diff` に渡す。これらは外部プロセスを呼ぶためテスト境界を切る（= 本体は薄く保ち、パーサ側でテスト）。
9. **空 tree SHA フォールバック**: `head_sha` の中で `git rev-parse HEAD` が失敗（exit code != 0）した場合、空 tree SHA `4b825dc642cb6eb9a060e54bf8d69288fbee4904` を返す。テストで `tempfile::tempdir()` + `git init` 直後のリポを作り、`head_sha` がフォールバックを返すこと、および `compute_diff` が全 worktree 内ファイルを `Added` 行として返すことを assert。
10. **untracked 合成**: `git status --porcelain` から `??` エントリを抽出し、`FileStatus::Untracked` として `FileDiff` に合成する。untracked の hunk は「ファイル全行が追加」として表現する（`std::fs::read_to_string` で内容を読んで `Added` 行に変換）。バイナリ判定は `std::fs::read` の先頭 8KB に NUL バイトがあるかで簡易的に行い、見つかれば `DiffContent::Binary`。

**このマイルストーン終了時の成果**: `cargo test --all-targets` で `src/git.rs` 内の単体テストが全て通る。手動で `cargo run -- --help` は通るが TUI はまだ起動しない。

### Milestone 2: `src/watcher.rs` — ファイル監視 + デバウンス + git_dir 解決

`notify-debouncer-full` の `new_debouncer` で 2 系統の watcher を張る。kizu 側の `.gitignore` フィルタは持たない (ADR-0005)。

1. **git_dir の解決**: M1 で追加した `git::git_dir(root) -> Result<PathBuf>` を使い、絶対 git ディレクトリパスを取得する。通常リポなら `<root>/.git`、linked worktree なら `<main_repo>/.git/worktrees/<name>` などになる。`watcher::start(root, git_dir)` の引数として渡す。
2. **worktree watcher**: git worktree のルート（`find_root` の戻り値）を `RecursiveMode::Recursive` で監視。デバウンス 300ms。
3. **HEAD watcher**: `<git_dir>/HEAD` と `<git_dir>/refs/` を監視。デバウンス 100ms。linked worktree でも正しく動く。
4. **git_dir プレフィックス除外**: worktree watcher のコールバックで受け取ったイベント絶対パスについて、`git_dir` のプレフィックスに一致するものは無条件で破棄する (feedback loop 回避)。これだけがハードコード除外。
5. **`.gitignore` フィルタは持たない**: ADR-0005 通り、`target/` などのノイズは git diff 自体が `.gitignore` 尊重で吸収するので、watcher 側でフィルタしない。`compute_diff` の結果がゼロなら `App.files` も画面も変わらない。
6. **tokio チャネル**: `tokio::sync::mpsc::unbounded_channel::<WatchEvent>()` で `Sender` を debouncer のコールバックに渡す。`UnboundedSender::send` は sync 文脈から呼べるため bridge 不要。イベント種別に応じて `WatchEvent::Worktree` または `WatchEvent::GitHead` を送る。`WatchHandle` は `UnboundedReceiver<WatchEvent>` とデバウンサの所有権（drop で停止）を保持する。
7. **背圧 / coalescing は app 層で実施**: watcher 側はバッファせず素直にイベントを流す。drain ベースの coalescing は M3 の `app::run` ループ側で実装する (ADR-0005)。

**テスト戦略**: notify 周りは I/O が絡むため Rust 統合テスト 2 本に留める（実時間挙動の検証は M6 の tuistory e2e に移管）。

- `watcher::tests::worktree_event_is_received` — `tempfile::tempdir()` で一時的な git リポジトリを作り、`watcher::start` 後にファイル書き換えを行い、`tokio::time::timeout(Duration::from_secs(2), watch_rx.recv())` で `WatchEvent::Worktree` を受信できることを assert
- `watcher::tests::git_dir_writes_are_excluded` — tempdir + git init 後に `<git_dir>/test_lock` 等を直接書き換え、`watch_rx.recv()` が **`Worktree` を返さない** ことを `tokio::time::timeout(500ms)` で assert

`#[tokio::test(flavor = "current_thread")]` 属性を使う。

**このマイルストーン終了時の成果**: `cargo test --all-targets` で watcher の統合テストが 2 本追加され、全テストが通る。

### Milestone 3: `src/app.rs` — 状態管理 + tokio イベントループ

1. **初期化**: `App::bootstrap(root)` を追加し、`git::head_sha(&root)` でベースラインを取得 (失敗時は空 tree SHA フォールバック)、`git::compute_diff(&root, &baseline)` で初期 `files` を埋める。`bootstrap` は **sync 関数** のまま実装する（async 伝染を抑える境界）。bootstrap 自体が失敗した場合は anyhow::Error を上に伝搬し、`main` が stderr に吐いて exit する (fail-fast)。
2. **イベントループ**: `app::run()` を `async fn` として実装し、`#[tokio::main(flavor = "current_thread")]` から呼び出す。
   - `watcher::start(&root)` で `WatchHandle` を取得（`events: tokio::sync::mpsc::UnboundedReceiver<WatchEvent>`）
   - `crossterm::event::EventStream::new()` でキー入力ストリームを作る
   - `tokio::select!` で 2 つのストリームを統合:

         loop {
             tokio::select! {
                 Some(Ok(crossterm_ev)) = events.next() => {
                     if let Event::Key(k) = crossterm_ev { app.handle_key(k); }
                 }
                 Some(first) = watch_rx.recv() => {
                     // drain ベースの coalescing (ADR-0005)
                     let mut worktree = matches!(first, WatchEvent::Worktree);
                     let mut head     = matches!(first, WatchEvent::GitHead);
                     while let Ok(more) = watch_rx.try_recv() {
                         match more {
                             WatchEvent::Worktree => worktree = true,
                             WatchEvent::GitHead  => head     = true,
                         }
                     }
                     if worktree { app.recompute_diff(); }  // Err は last_error に格納、? は使わない
                     if head     { app.mark_head_dirty(); }
                 }
             }
             terminal.draw(|f| ui::render(f, &app))?;
             if app.should_quit { break; }
         }

   - **drain coalescing**: 1 件のイベントを受信したら同じターン内に積まれている残りを `try_recv` ループで全部吸い上げ、`Worktree` フラグが立っていれば `compute_diff` を **1 回だけ** 呼ぶ。これにより 50000 件のバーストイベント (例: `cargo clean && cargo build`) も 1 回の `compute_diff` 呼び出しに吸収され、キューが伸び続けることがない。
   - `WatchEvent::Worktree` 受信時: `App::recompute_diff` の中で `git::compute_diff` (sync) を再実行する。`Ok(files)` なら `App.files = files` に置き換え、`last_error = None` をクリアし、フォローモードが ON なら最後に変更されたファイルへセレクションを移動。`Err(e)` なら `last_error = Some(e.to_string())` に記録し `files` は前回成功時の状態を保持する。
   - `WatchEvent::GitHead` 受信時: 「HEAD が動いた」旨のインジケータを更新するのみ（ベースラインは `R` 手動操作でのみ再取得）
3. **キーバインド**:
   - `q` / `Ctrl-C`: 終了
   - `j` / `↓`: ファイル選択 +1、フォローモードを `manual` にダウングレード
   - `k` / `↑`: ファイル選択 -1、フォローモードを `manual` にダウングレード
   - `Ctrl-d`: 右ペイン diff を半ページ前進、フォローモードを `manual` にダウングレード
   - `Ctrl-u`: 右ペイン diff を半ページ後退、フォローモードを `manual` にダウングレード
   - `g`: 右ペイン diff を先頭にスクロール、フォローモードを `manual` にダウングレード
   - `G`: 右ペイン diff を末尾にスクロール、フォローモードを `manual` にダウングレード
   - `f`: フォローモード復帰（自動スクロールと自動セレクションが再開）
   - `R` (Shift+r): `baseline_sha = head_sha()` を即座に再取得し、`compute_diff` で `files` を再計算する。1 打で確定。
4. **パス追跡セレクション**: `App.selected_path: Option<PathBuf>` を保持し、`recompute_diff` の終わりで新 `files` から `position(|f| Some(&f.path) == selected_path.as_ref())` でインデックスを再計算する。見つからない場合は前回の `selected` インデックスを `min(files.len() - 1)` で clamp してフォールバック。これにより watcher イベントで他ファイルが追加/削除されてもユーザーが「見ているファイル」がずれない。
5. **mtime ソートと follow ターゲット**: `recompute_diff` 内で `compute_diff` の戻り値の各 `FileDiff` に `std::fs::metadata(&f.path).and_then(|m| m.modified()).unwrap_or(UNIX_EPOCH)` で `mtime` を埋め、`files.sort_by(|a, b| b.mtime.cmp(&a.mtime))` で降順ソート。フォローモード ON のときは `files[0]` (= mtime 最新) を `selected_path` にする。
6. **テスト**: 純粋なキー処理関数 `App::handle_key(&mut self, key: KeyEvent)` を切り出し、`j` でセレクションが進む / `f` でフォローが戻る / `Ctrl-d` で diff_scroll が増える / `R` で baseline 再取得フラグが立つ等を単体テストする。`recompute_diff` のパス追跡 / mtime ソート / エラー保持の振る舞いも `#[cfg(test)]` で小さな fixture を使って単体テストする。イベントループ本体（ratatui terminal loop）は M6 の tuistory e2e で検証。

### Milestone 4: `src/ui.rs` — ratatui レイアウト + シンタックスハイライト + バイナリ + エラー表示

1. **レイアウト**: `Layout::default().direction(Direction::Vertical)` で上下分割し、上段を `Layout::default().direction(Direction::Horizontal).constraints([Constraint::Percentage(30), Constraint::Percentage(70)])` で左右分割、下段は `Constraint::Length(1)` のフッター。
2. **左ペイン**: `List` ウィジェットで `App.files` を描画。各行は `"{status} {path} +{added}/-{deleted}"`（例: `"M src/auth.rs +12/-3"`）。`DiffContent::Binary` のファイルは `+N/-M` の代わりに `bin` を表示（例: `"M assets/icon.png  bin"`）。選択行は反転色。
3. **右ペイン (テキスト)**: `Paragraph` または `List` で選択ファイルの hunks を描画。各 `DiffLine` を `LineKind` に応じて色分け（added=緑 / deleted=赤 / context=無色）。
4. **右ペイン (バイナリ)**: `DiffContent::Binary` の場合は単一行 `[binary file - diff suppressed]` を中央寄せで表示し、hunks は描画しない。
5. **シンタックスハイライト**: `syntect::easy::HighlightLines` でファイル拡張子から syntax を決定し、hunk 行のコンテンツ部分にハイライトを乗せる。色情報は `ratatui::style::Color::Rgb` に変換。バイナリは対象外。
6. **フッター**: `"[follow] <current_file> +A/-D | session: +ΣA/-ΣD N files"` の形式。フォローモード OFF 時は `[follow]` を `[manual]` に切り替える。`App.last_error` が `Some(msg)` のときはフッタ右端に赤色で `× {msg}` を追記する（長すぎる場合は ellipsis で truncate）。
7. **大規模 diff の truncate**: テキスト diff の合計行数が 2000 を超えるファイルは末尾を切り捨て、最終行に `[+N more lines truncated]` を `Color::DarkGray` で表示する。truncate は `App::visible_diff_lines(file_idx)` のような ヘルパメソッドで行い、`FileDiff` 自体には全行を保持する（v0.1.1 の「展開キー」実装の素地）。
8. **空状態 (No changes)**: `App.files.is_empty()` の場合、左右どちらのペインも個別ウィジェットを描かず、上半分全体に中央揃えで `No changes since baseline (HEAD: <short_sha>)` を 1 行で表示する。`<short_sha>` は `App.baseline_sha[..7]`。フッタは `[follow] -- | session: 0 files`。
9. **パフォーマンス**: 初版は毎フレーム全ファイル再描画でよい。syntect のハイライトは重いので、選択ファイルのみハイライト計算を行う（他のファイルは行数カウントのみ）。
10. **diff スクロール状態**: `App.diff_scroll: usize` を「右ペイン上端の論理行オフセット」として保持。`Ctrl-d` は `+= viewport_height / 2`、`Ctrl-u` は `-= viewport_height / 2` (saturating)、`g` は `0`、`G` は `(visible_lines - viewport_height).max(0)`。follow ON で `recompute_diff` 後に新しい file の最後の hunk の `new_start` 相当の行までスクロールする。`render` は `diff_scroll` を起点に行を切り出して描画する。

**ratatui は毎フレーム再構築が前提なので、`render` はイミュータブルな `&App` から純粋に `Frame` を描画する関数として実装する。**

## Idempotence and Recovery

- **Red/Green/Refactor サイクルは完全にべき等**: 各サイクルは `git` リポジトリの worktree のみを変更し、`.git` や外部ファイルには触れない。失敗したら `git checkout -- src/` で巻き戻せる。
- **watcher のテスト**: `tempfile::tempdir()` は Drop で自動削除されるため、テストが異常終了してもゴミが残らない。万一 `/tmp/.tmp*` が残った場合は手動で削除可能。
- **TUI 異常終了時の端末復旧**: `std::panic::set_hook` で panic 時に `disable_raw_mode` + `LeaveAlternateScreen` を実行する。それでも端末が壊れた場合は `reset` コマンドで復旧する。
- **ベースライン再取得（`R`）**: 失敗しても副作用なし（state のみ書き換え）。誤爆しても次の `R` で現在の HEAD に再固定するだけ。
- **CI 相当の検証**: 何度実行しても同じ結果になる。`cargo clean` は不要。

### 依存理由

- **notify-debouncer-full を直接 notify ではなく使う理由**: SPEC.md 指定の 300ms / 100ms デバウンスを自前実装するより、公式ラッパの方がバーストイベントをマージする際のエッジケース（rename / atomic replace / 一時ファイル）を正しく扱う。
- **kizu 側に `.gitignore` フィルタを持たない理由**: ignore crate API の正しい使い方が `Gitignore::new()` だけでは不十分で、ネスト `.gitignore` 対応の `dir::Ignore` が `pub(crate)` のため自前実装になる。コストの方が「フィルタを持たないことで `compute_diff` が無駄に呼ばれる」コストより高い。drain coalescing を入れれば 50000 件のバーストも 1 回の `compute_diff` 呼び出しに吸収される。詳細は ADR-0005。
- **`git rev-parse --absolute-git-dir` で git ディレクトリを解決する理由**: `<root>/.git` ハードコードでは linked worktree が動かない。詳細は ADR-0005。
- **drain coalescing を app 層に置く理由**: `unbounded_channel` を 1 件ずつ消費するとバーストでキューが伸び続けキー応答が遅延する。`try_recv` ループで残りを吸い上げて 1 回 `compute_diff` する設計が最小行数で背圧を実装できる。詳細は ADR-0005。
- **tokio の current_thread flavor を選ぶ理由**: TUI に並列実行は不要。`Send` 制約を持ち込まずに済むため `Rc` / `RefCell` / `&mut self` の自由度が増す。
- **syntect を ratatui のスタイリングに変換する理由**: bat / delta と同じエンジンなので言語サポートが広く、追加メンテ不要。
- **git CLI shell out を維持する理由**: CLAUDE.md / SPEC.md の明示方針。`--no-renames` 等を再実装する労力を回避し、git の挙動と必ず一致させる。
- **tuistory を採用する理由**: kizu の価値 (visual diff observation) を直接守れる唯一の e2e 手段。詳細は ADR-0004。
