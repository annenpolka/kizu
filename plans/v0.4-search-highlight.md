# v0.4 `/` 検索: 一致単語を本文側でハイライトする

この ExecPlan はリビングドキュメントです。`Progress`、`Surprises & Discoveries`、`Decision Log`、`Outcomes & Retrospective` の各セクションは作業の進行に合わせて随時更新してください。

## Purpose / Big Picture

kizu の `/` スラッシュ検索はすでに smart-case 検索 + `n`/`N` ジャンプを実装しているが、**一致した単語そのものは視覚的に強調されない**。ユーザは `n`/`N` でカーソル行に飛べるだけで、「行のどこに一致があるか」「同じ画面に他にも一致があるか」をテキストから目で探す必要がある。

v0.4 のこの追補ではエディタ的な in-body ハイライトを導入する:

- **全一致**: 本文描画時に、一致バイト範囲へ下線 (`UNDERLINED`) + 太字 (`BOLD`) を overlay。既存の syntax HL の fg と diff の bg は保持する
- **現在一致 (`current`)**: `n`/`N` が指している一致のみ bg を Yellow、fg を Black、Bold で **反転強調**。画面内に 1 箇所だけになる
- 一致は `Enter` で query を確定した瞬間だけでなく、その後の watcher-driven な `recompute_diff` 後も追従する (layout の row index が変わるので `SearchState.matches` を再計算する)

非対象:

- `search_input` (コンポーザ中 = query 編集中) のインクリメンタルハイライトは v0.4 スコープ外
- file view (`Enter`) の内部ハイライトは既存挙動のまま (search state は scroll view 専用)

ユーザ観点の成果:

- `/foo<Enter>` で全ての `foo` に下線が付き、`n` で順に現在一致が Yellow bg で強調される
- 並走する AI エージェントが編集して watcher が再計算しても、同じ query のハイライトが自動追従

動作確認:

1. 適当なリポジトリで kizu を起動し、同じ単語 (例: `pub`) を含む hunk を 2 つ以上表示
2. `/pub<Enter>` で確定
3. 画面内のすべての `pub` に下線が付き、最初の一致が Yellow bg + 黒文字で強調表示される
4. `n` を押すごとに Yellow bg が次の一致に移動、`N` で逆方向に移動、末尾/先頭で wrap-around
5. 別プロセスで該当ファイルを編集して watcher 再計算が走った後も、残っている一致にハイライトが残り続ける

## Surprises & Discoveries

- `apply_computed_files` は private method なので、ui.rs 側のテストから呼べなかった。rehydrate のテストは app.rs 側の `mod tests` に配置した (render のテストだけ ui.rs)。境界が素直に分かれて結果的にきれい
- clippy の `doc_lazy_continuation` が `/// -` のようなリストっぽい段落を嫌う。RowRenderCtx の doc は箇条書きをやめて関数参照に置き換えた
- syntect fallback path (`tokens.len() == 1 && fg == Color::Reset`) の扱いが元の render_diff_line とは分離され、`per_char_fg` ヘルパー内で集約された。結果、nowrap / wrap の両経路で fallback 扱いが一致した (元は nowrap だけ別経路)

## Decision Log

- Decision: current match = Yellow bg + Black fg + Bold、other match = UNDERLINED + BOLD に下線のみ
  Rationale: diff の bg (`bg_added` 緑 / `bg_deleted` 赤) と syntax HL の多彩な fg を崩さず、かつ現在一致だけは強いコントラスト (Yellow/Black) で反転させる。下線はターミナル互換性が高く、syntax fg を潰さない。Yellow は既存 `[search]` footer と同じ系統色で UI コヒーレンス。
  Date/Author: 2026-04-20 / Claude

- Decision: recompute 後は `SearchState.query` を再実行して `matches` を作り直す (= rehydrate)
  Rationale: `MatchLocation.row` は layout row index ベースで、`build_layout` で容易に無効化する。`SearchState` を丸ごと drop してしまうと watcher 再計算のたびに `n`/`N` が失われる。query は保持し、matches だけ再計算して `current` を `min(current, matches.len()-1)` にクランプするのが最小侵襲。
  Date/Author: 2026-04-20 / Claude

- Decision: ハイライト overlay は per-char attribute (fg + match_kind) を一度組み立ててから span に groupBy する方式
  Rationale: 既存 `render_diff_line` は syntax token 単位で `take_cells` を呼んで span を作るが、そこに「バイト範囲の overlay」を差し込むと、トークン境界 × match 境界 × cell budget 境界の 3 軸で細切れになる。char 単位で属性を一度平らに並べて groupBy する方が論理的に単純で、CJK の cell width 計算とも分離できる。
  Date/Author: 2026-04-20 / Claude

- Decision: t-wada 流 TDD を遵守 (Red 先行)
  Rationale: `render_diff_line` / `render_diff_line_wrapped` の修正は広い面積に触れるので、先に「current match の cell bg が Yellow になる」「other match の cell が UNDERLINED になる」ことを TestBackend で assert する。旧実装で FAILED を確認してから Green に入る。
  Date/Author: 2026-04-20 / Claude

```rust
    .map(|s| s.matches.iter().enumerate()
        .filter(|(_, m)| m.row == row_idx)
        .map(|(i, m)| (m.byte_start, m.byte_end, i == s.current))

```

### 3. Green: `render_diff_line` / `render_diff_line_wrapped` を per-char-attr 方式に書き換え

両関数の共通部分をヘルパーとしてくくり出す:

```rust
#[derive(Clone, Copy)]
enum HlKind { None, Other, Current }

fn build_char_attrs(
    content: &str,
    tokens: Option<&[HlToken]>,
    matches: &[(usize, usize, bool)],
) -> Vec<(Color, HlKind)> {
    // 1. token ベースで (fg, None) を char ごとに敷き詰める
    // 2. matches の byte range が char の byte offset を含むなら kind を上書き
}

fn style_for_attr(base: Style, attr: (Color, HlKind)) -> Style {
    match attr.1 {
        HlKind::Current => Style::default()
            .bg(Color::Yellow)
            .fg(Color::Black)
            .add_modifier(Modifier::BOLD),
        HlKind::Other => base.fg(attr.0)
            .add_modifier(Modifier::UNDERLINED | Modifier::BOLD),
        HlKind::None => base.fg(attr.0),
    }
}
```

`render_diff_line` (nowrap): chars を `take_cells` 的にループしつつ、同一 attr の run を 1 span にまとめる。`body_width` に達したら truncate、残りは `base_style` で pad。

`render_diff_line_wrapped`: `wrap_at_chars` で作った chunk を走りながら、各 chunk 内の chars について同様に run をまとめる。char_offset を global にトラックして matches の byte range に対応付ける。

### 4. Green: `apply_computed_files` で SearchState の rehydrate

`src/app.rs` の `apply_computed_files` 末尾 (layout rebuild 後) に以下を追加:

```rust
if let Some(state) = self.search.as_mut() {
    let query = state.query.clone();
    state.matches = find_matches(&self.layout, &self.files, &query);
    if state.matches.is_empty() {
        state.current = 0;
    } else {
        state.current = state.current.min(state.matches.len() - 1);
    }
}
```

`state.matches.is_empty()` のとき `current` を 0 に戻すのは、元 matches が無くなった後にもう一度一致が復活したとき `n` 押下で正しく最初に飛べるようにするため。
